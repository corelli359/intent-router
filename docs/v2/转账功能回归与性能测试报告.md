# 转账功能回归与性能测试报告

## 1. 测试目标与范围

- 仅围绕当前唯一具备提槽能力的意图 `transfer_money` 做回归，不新增任何新意图。
- 覆盖以下场景：
  - 单轮意图识别与提槽
  - 多轮意图识别与提槽
  - 多意图下各意图单轮识别与提槽
  - 多意图下各意图多轮提槽
  - 同意图穿插/拆分后的识别与提槽恢复
- 在功能回归后，补充 `router_only` 基线性能测试，评估当前服务可承载并发及 CPU/内存表现。

## 2. 测试环境

### 2.1 功能回归环境

- 日期：`2026-04-18`
- 分支：`test/v3-concurrency-test`
- Router：`http://127.0.0.1:8012`
- Transfer agent：`http://127.0.0.1:8021/api/agent/run`
- catalog：`.local-data/router-transfer-only-catalog`
- 挡板：
  - `ROUTER_LLM_BARRIER_ENABLED=false`
  - `ROUTER_AGENT_BARRIER_ENABLED=false`
- 说明：
  - 该环境走真实本地 HTTP 链路，依赖外部 LLM 完成识别和提槽。
  - 自动断言只对稳定项做硬校验：`intent_code == transfer_money`，以及必填槽位 `payee_name`、`amount`。
  - 可选槽位在真实 LLM 链路下存在波动，本轮不作为自动回归硬断言条件。

### 2.2 性能测试环境

- Router：`http://127.0.0.1:8013`
- catalog：`.local-data/router-transfer-only-catalog`
- 挡板：
  - `ROUTER_LLM_BARRIER_ENABLED=true`
  - `ROUTER_AGENT_BARRIER_ENABLED=true`
- 执行模式：`executionMode=router_only`
- 压测输入：`转 500 给王芳的卡`
- 说明：
  - 该输入能在无模型启发式回退路径下稳定识别为 `transfer_money` 并进入 `ready_for_dispatch`。
  - 像 `给小红转200` 这类自然表达，在当前 barrier fallback 下会出现候选意图但不一定进入 `ready_for_dispatch`，因此不适合作为压测基线。
  - 本轮性能测试是本地直连 router API 的基线测试，不包含真实模型 I/O 和真实 agent I/O。

## 5. 未通过项判定

本轮自动脚本中的 3 个未通过项，不能简单等价成产品回归。需要拆成两类看。

### 5.1 环境限流导致的未验证

本轮功能回归依赖真实外部 LLM。Router 日志中明确出现以下错误：

- `openai.RateLimitError`
- `limit_type: concurrent_sessions`
- `current: 15`
- `limit: 15`

在该错误出现时，router 会返回：

- `current_graph = null`
- `pending_graph = null`
- assistant 文案为 `当前意图识别服务繁忙，请稍后重试。`

因此，本轮自动回归中以下两个场景属于“受外部模型并发上限影响，未能稳定验证”，不能直接判定为产品逻辑失败：

- `multi_intent_single_turn_two_transfers`
- `multi_intent_multi_turn_two_transfers`

### 5.2 当前真实存在的功能缺口

同分支、同真实链路下的补充复测显示：

- 多意图单轮：
  - `先给妈妈转500，再给弟弟转600`
  - 可以识别成两个 `transfer_money` 节点
  - `pending_graph.status = waiting_confirmation`
  - 确认后两个节点都能完成
- 多意图多轮：
  - `先给妈妈转账，再给弟弟转账`
  - 可识别两个转账节点
  - 确认后分别补 `500`、`600`
  - 最终可完成
  - 但真实链路耗时很高，并且容易被外部 LLM `429` 打断
- 同意图穿插/恢复：
  - `帮我给妈妈转账`
  - `算了，先给弟弟转200`
  - `继续刚才给妈妈那个，300`
  - 当前不会挂起“妈妈转账业务”再恢复，而是会直接改写当前业务，第三轮也不能正确恢复到原业务对象

因此，当前可以给出如下判定：

- 单轮转账识别与提槽：通过
- 多轮转账识别与提槽：通过
- 多意图单轮：能力存在，但本轮自动回归受 LLM 并发限流影响，未稳定自动化
- 多意图多轮：能力存在，但本轮自动回归受 LLM 并发限流影响，且真实链路耗时偏高
- 同意图穿插/恢复：当前不通过，属于真实能力缺口

## 7. 阶梯压测结果

### 7.1 低到中等并发

| 并发 | 成功率 | RPS | Avg | P95 | Max |
| --- | --- | --- | --- | --- | --- |
| `1` | `100%` | `171.35` | `5.83 ms` | `7.09 ms` | `123.29 ms` |
| `5` | `100%` | `262.33` | `19.03 ms` | `25.78 ms` | `33.57 ms` |
| `10` | `100%` | `267.29` | `37.29 ms` | `43.21 ms` | `127.07 ms` |
| `20` | `100%` | `270.44` | `73.46 ms` | `89.39 ms` | `179.10 ms` |
| `30` | `100%` | `247.66` | `119.95 ms` | `207.05 ms` | `666.09 ms` |
| `50` | `100%` | `241.63` | `203.56 ms` | `363.56 ms` | `888.69 ms` |
| `80` | `100%` | `227.31` | `342.07 ms` | `789.00 ms` | `1620.79 ms` |
| `120` | `100%` | `210.20` | `547.68 ms` | `963.44 ms` | `2384.87 ms` |

观察：

- 吞吐峰值出现在 `20` 并发附近，约 `270 RPS`。
- `30` 并发以后，吞吐不再提升，开始明显换成延迟增长。
- 若以 `P95 < 1s` 为准，当前可认为 `120` 并发仍在可接受边界内。
- 若以 `P95 < 500ms` 为准，推荐控制在 `50` 并发以内。

### 7.2 高并发

- 虽然仍然 `0` 失败，但本质上已经进入明显排队状态。

### 7.3 极限并发

| 并发 | 成功率 | RPS | Avg | P95 | Max |
| --- | --- | --- | --- | --- | --- |
| `800` | `100%` | `18.47` | `33258.09 ms` | `34653.43 ms` | `34996.89 ms` |
| `1000` | `100%` | `84.36` | `9477.60 ms` | `15565.55 ms` | `15833.46 ms` |
| `1500` | `100%` | `123.43` | `8653.13 ms` | `13574.81 ms` | `14253.10 ms` |
| `2000` | `100%` | `106.92` | `12799.61 ms` | `21187.64 ms` | `21645.42 ms` |

观察：

- 到 `2000` 并发为止，脚本未出现 HTTP 失败或状态失败。
- 但 `800+` 并发时，延迟已经非常高，`800` 并发平均延迟超过 `33s`。
- 该结果只能说明“当前单进程 router_only 基线在本机上没有被压垮”，不能说明“2000 并发适合生产”。

## 8. CPU 与内存

压测过程中用 `ps` 对 router 进程做采样，综合三轮压测结果：

- 峰值 CPU：约 `40.7%`
- 峰值 RSS：约 `227.47 MB`

说明：

- 当前 `router_only + 双挡板 + transfer-only catalog` 基线内存占用并不高。
- 在本轮单进程本地测试中，瓶颈更像是排队与事件循环处理时延，而不是先被内存打满。
- 如果后续要找真实上限，应继续结合：
  - 多 worker 部署
  - 连接池配置
  - session 对象生命周期
  - admin 发起时刻计时
  - Kubernetes SVC 链路
  - 实时 CPU/内存监控

## 9. 结论

### 9.1 功能结论

- 当前仅针对 `transfer_money` 做回归，单轮与多轮提槽能力通过。
- 多意图单轮、多意图多轮在真实链路中具备跑通能力，但本轮自动回归受外部 LLM `concurrent_sessions=15` 限流影响，未稳定自动化。
- 同意图穿插/恢复当前不通过，这是现阶段真实能力缺口。

### 9.2 性能结论

- 在 `router_only + LLM barrier + agent barrier` 基线下，本轮已验证到 `2000` 并发仍无失败。
- 但从延迟看，真正有参考价值的并发区间要按 SLA 解读：
  - 若目标 `P95 < 500ms`，建议不高于 `50`
  - 若目标 `P95 < 1s`，建议不高于 `120`
  - 若只追求“0 失败返回”，当前本机单进程已验证到 `2000`
- 当前吞吐峰值大致出现在 `20` 并发附近，约 `270 RPS`。
