# 层级路由首批实施设计

状态：进行中

更新时间：2026-04-11

## 1. 首批范围

本轮代码改造只落地主链路骨架，不做完整条件治理，不做完整 Router 侧槽位治理。

本轮交付范围：

1. Intent 注册模型补齐层级字段
2. Admin/Router 持久化与 API 透传层级字段
3. Router Catalog 提供 domain 视图
4. 新增 `DomainRouter`
5. 新增 `LeafIntentRouter`
6. 新增 `HierarchicalIntentRecognizer`
7. Router runtime 增加 `hierarchical` 理解模式开关
8. `hierarchical` 模式下首轮消息走“domain -> leaf -> planner”
9. 补基础测试

本轮明确不做：

1. 条件提取器
2. 条件校验器
3. 条件自动补图
4. SlotExtractor
5. SlotValidator
6. Router pre-dispatch gate
7. 结构化 memory

## 3. 架构设计

## 3.1 当前首批架构

本轮不把 graph runtime 拆开重做，而是在理解链路最前端增加一层分层路由。

```text
用户消息
-> HierarchicalIntentRecognizer
   -> DomainRouter
   -> LeafIntentRouter
-> RecognitionResult(leaf intents)
-> 现有 planner
-> 现有 graph runtime
-> agent
```

这样做的原因：

1. 改动面小
2. 不破坏现有 graph node / task / dispatch 协议
3. 首批可以先把“全量扁平 intent 竞争”替换掉
4. 后续接 SlotExtractor / Validator 时还有清晰演进空间

## 3.2 核心模块关系

### Intent Registry

负责：

- 存储 domain 和 leaf intent 的层级字段

不负责：

- 真正执行 domain routing

### RepositoryIntentCatalog

负责：

- 提供 active leaf intents
- 提供 active domains 视图

### DomainRouter

负责：

- 在 domain 层做第一次收窄

输入：

- message
- domains
- recent_messages

输出：

- `primary_domains`
- `candidate_domains`

### LeafIntentRouter

负责：

- 在某个 domain 的 leaf intents 中继续识别

输入：

- message
- leaf intents in domain

输出：

- `RecognitionResult`

### HierarchicalIntentRecognizer

负责：

- 编排 domain -> leaf 两段路由
- 对外仍保持 `IntentRecognizer` 接口

### Existing Planner / Graph Runtime

负责：

- 消费 leaf intent 识别结果
- 继续按现有逻辑生成 graph 和运行 graph

## 4. 旅程设计

## 4.1 单域单叶子

用户输入：

```text
给我弟弟转 500
```

旅程：

1. DomainRouter 输出 `transfer`
2. Catalog 发现 `transfer` 域下只有一个 leaf intent：`transfer_money`
3. LeafIntentRouter 直接命中 `transfer_money`
4. planner 根据 leaf intent 继续建图

特点：

- 不需要在全量 intent 里竞争
- 不需要 leaf 阶段再调用 LLM

## 4.2 单域多叶子

用户输入：

```text
帮我交燃气费
```

旅程：

1. DomainRouter 输出 `payment`
2. Catalog 返回 `payment` 域下的多个 leaf intents
3. LeafIntentRouter 在 `pay_electricity / pay_water / pay_gas` 子集中识别
4. 命中 `pay_gas_bill`
5. planner 建图

特点：

- 只在 payment 域子集内识别
- 相比全量扁平 intent 竞争，误匹配更少

## 4.3 多域并存

用户输入：

```text
先查余额，再给妈妈转 500
```

本轮首批实现只保证：

1. DomainRouter 可以返回多个 domain
2. LeafIntentRouter 可以分别收敛到多个 leaf intents
3. planner 继续按既有逻辑处理顺序关系

说明：

- 条件与复杂图关系的精细治理是后续阶段
- 首批先解决“多域收敛到正确 leaf”问题

## 5. 时序设计

## 5.1 Flat Mode

```text
Client -> Router API
Router API -> Flat Recognizer
Flat Recognizer -> LLM
Flat Recognizer -> Router API
Router API -> Planner
Planner -> Router API
Router API -> Graph Runtime
Router API -> Agent
```

## 5.2 Hierarchical Mode

```text
Client -> Router API
Router API -> HierarchicalIntentRecognizer
HierarchicalIntentRecognizer -> DomainRouter
DomainRouter -> LLM
DomainRouter -> HierarchicalIntentRecognizer
HierarchicalIntentRecognizer -> LeafIntentRouter
LeafIntentRouter -> LLM or direct single-leaf shortcut
LeafIntentRouter -> HierarchicalIntentRecognizer
HierarchicalIntentRecognizer -> Router API
Router API -> Planner
Planner -> Router API
Router API -> Graph Runtime
Router API -> Agent
```

## 5.3 回退时序

当 hierarchical 路由失败时：

```text
HierarchicalIntentRecognizer
-> DomainRouter 无结果 或 leaf route 无结果
-> fallback flat recognizer
-> RecognitionResult
```

目标：

- 不让新链路直接导致可用性下降
- 首批上线时以保守兼容为先

## 6. 首批实现原则

1. 不改 graph node 结构
2. 不改 agent 协议
3. 不改现有 planner 语义
4. 新链路失败时回退 flat recognizer
5. 保持配置可切换

## 7. 验收口径

首批完成后，应满足：

1. Admin 可注册 domain 字段
2. Router Catalog 可列出 active domains
3. `hierarchical` 模式可根据 domain 缩小 leaf intent 搜索空间
4. 单 leaf domain 可直接命中 leaf intent
5. 多 leaf domain 可在子集内识别
6. 空结果可回退 flat recognizer
7. 现有 graph runtime 与 agent dispatch 不被破坏
