## 1. 本次首期开发范围

本次首期开发只实现主链路骨架，不在这一轮落地条件专项治理。

本次范围：

1. 意图注册模型支持 domain 维度
2. Catalog 提供按 domain 聚合视图
3. Router 增加 hierarchical understanding mode
4. Router 新增 `DomainRouter`
5. Router 新增 `LeafIntentRouter`
6. Router 新增 `HierarchicalIntentRecognizer`
7. Runtime 装配 `flat|hierarchical` 双轨切换
8. 补最小回归测试

本次明确不做：

1. `SlotExtractor`
2. `slot_validator`
3. Router pre-dispatch gate
4. `ConditionExtractor`
5. `condition_validator`
6. 条件自动补图
7. 条件 Router 追问
8. 结构化 memory store 全量重构

## 3. 架构设计

## 3.1 总体架构

首期目标架构：

```text
Intent Registry
  -> Admin API / Repository
  -> Router Catalog Snapshot

User Message
  -> DomainRouter
  -> LeafIntentRouter
  -> SlotExtractor
  -> SlotValidator
  -> GraphCompiler
  -> GraphRuntime / Orchestrator
  -> Agent
```

## 3.2 模块职责

### Intent Registry

职责：

- 存储意图注册定义
- 提供 domain 维度信息
- 仍由 Admin 作为 source of truth

### Catalog Snapshot

职责：

- 从仓储拉取 active intents
- 构建按 domain 聚合的只读快照
- 为 Router 提供快速读视图

### DomainRouter

职责：

- 只判断大类
- 返回 primary / candidate domains
- 不输出叶子槽位

### LeafIntentRouter

职责：

- 只在 domain 的子集中判断 leaf intent
- 返回 leaf intent 与 source fragment

### SlotExtractor

职责：

- 基于 leaf intent 的 `slot_schema` 抽取执行槽位
- 输出 `slot_memory`、`slot_bindings`、`missing_slots`

### SlotValidator

职责：

- required 槽位检查
- slot key/value type 检查
- source_text / grounding 检查
- history 候选受控补全入口预留

### GraphCompiler

职责：

- 把已确定的 leaf intents 组织为执行 graph
- 当前阶段只处理叶子节点与顺序关系

### Orchestrator

职责：

- 接管 graph 生命周期
- 在 task 创建前执行 validator gate
- 缺槽位时停在 Router

## 4. 用户旅程设计

## 4.1 单 domain 单 leaf

输入：

```text
帮我交燃气费 88 元
```

旅程：

1. DomainRouter 输出 `payment`
2. LeafIntentRouter 在 `payment` 域内输出 `pay_gas_bill`
3. SlotExtractor 抽取 `amount=88`
4. SlotValidator 发现缺 `gas_account_number`
5. Router 进入 `WAITING_USER_INPUT`
6. 用户补充户号
7. Router 只重跑当前 leaf intent 的 slot extraction / validation
8. 通过后编译 graph 并下发 agent

## 4.2 单 domain 单 leaf 直达执行

输入：

```text
给弟弟转 500
```

旅程：

1. DomainRouter 输出 `transfer`
2. LeafIntentRouter 输出 `transfer_money`
3. SlotExtractor 抽取 `recipient_name=弟弟`、`amount=500`
4. SlotValidator 判断如果仍缺卡号等关键参数，则进入 Router 追问
5. 若参数齐全，则 graph 生成单节点并下发 agent

## 4.3 多 domain 多 leaf

输入：

```text
先查余额，再给妈妈转 500
```

首期旅程：

1. DomainRouter 输出 `account_query`、`transfer`
2. LeafIntentRouter 输出 `query_account_balance`、`transfer_money`
3. SlotExtractor 分别为两个 leaf intent 抽取槽位
4. SlotValidator 校验各节点参数
5. GraphCompiler 依据 source fragment 与默认顺序策略生成两个叶子节点的执行图
6. Orchestrator 驱动执行

说明：

- 首期只支持显式顺序和默认顺序
- 条件阈值型条件放到后续阶段

## 5. 时序设计

## 5.1 首期正常时序

```text
User
-> Router API
-> GraphCompiler
-> DomainRouter
-> LeafIntentRouter
-> SlotExtractor
-> SlotValidator
-> GraphCompiler
-> Orchestrator
-> Agent
-> Orchestrator
-> User
```

详细说明：

1. Router API 接收消息
2. GraphCompiler 调用 `DomainRouter`
3. `DomainRouter` 返回一个或多个 domain
4. GraphCompiler 对每个 domain 调用 `LeafIntentRouter`
5. 对每个 leaf intent 调用 `SlotExtractor`
6. 对抽取结果调用 `SlotValidator`
7. 如果校验通过，GraphCompiler 产出 graph
8. Orchestrator 在 pre-dispatch gate 复核后创建 task
9. Agent 执行并返回流式结果

## 5.2 缺槽位时序

```text
User
-> Router API
-> DomainRouter
-> LeafIntentRouter
-> SlotExtractor
-> SlotValidator
-> Router WAITING_USER_INPUT
-> User supplies missing slot
-> SlotExtractor
-> SlotValidator
-> GraphCompiler
-> Agent
```

关键点：

- 缺槽位时不创建 task
- 用户补充后只重跑当前 leaf intent
- 不重新做整句全量理解

## 8. 验收标准

首期完成后，至少满足以下标准：

1. Intent 可带 domain 信息注册和查询
2. Router 能按 domain 收窄 leaf intent 集合
3. 单 leaf intent 的槽位抽取独立完成
4. 缺槽位时 Router 直接等待用户补充，不先下发 agent
5. Agent 不再是第一层发现缺槽位的地方
6. 原有 unified/legacy 模式仍保留兼容开关
