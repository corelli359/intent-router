# router-api 阶梯压测实施方案

## 1. 背景

当前仓库已经具备以下基础能力：

- `router-api`：负责意图识别、图编排、执行流转
- `admin-api`：负责管理端后端能力
- `admin-web`：负责管理端前端页面
- `k8s/intent/`：负责现有部署资源

本次目标是在 `test/v3-concurrency-test` 分支上补齐一条独立的性能测试链路，用于对 `router-api` 进行阶梯压测，并尽量排除真实大模型调用和浏览器网络波动对结果的干扰。

## 2. 目标与约束

### 2.1 目标

实现一套可在管理端发起的 `router-api` 阶梯压测方案，支持：

- 独立测试目标服务
- 阶梯式并发压测
- 可配置测试用例
- 可视化展示测试过程和结果
- 服务端发压，避免浏览器侧网络噪音

### 2.2 约束

- 压测必须增加大模型访问挡板，屏蔽真实模型 I/O
- 新增独立 Deployment，名称为 `router-api-test`
- 测试必须由 `admin` 服务端发起
- `admin` 发压时必须通过集群内 `svc` 访问 `router-api-test`
- 前端需要提供专业的性能测试页面

## 3. 总体方案

整体调用链路如下：

`admin-web -> admin-api -> router-api-test.intent.svc.cluster.local`

设计原则如下：

- 浏览器不直接请求 `router-api-test`
- 压测任务由 `admin-api` 统一调度和执行
- `router-api-test` 与正式 `router-api` 彻底隔离
- 第一阶段优先测试 router 会话、编排、状态流转能力，不引入真实 LLM I/O

## 4. router-api 大模型挡板方案

### 4.1 目标

在压测环境中禁止 `router-api-test` 发起真实大模型调用，使性能结果聚焦在以下部分：

- FastAPI 接口处理
- Session 创建与读取
- Graph 编排
- 状态流转
- SSE 或普通响应生成

### 4.2 设计

建议在 `router-service` 中新增压测挡板配置：

- `ROUTER_LLM_BARRIER_ENABLED=true`

挡板启用后：

- 所有真实 `llm_client.run_json(...)` 路径禁止访问外部模型
- 若压测路径误触发真实 LLM 调用，应立即失败并记录明确错误
- 压测主路径默认使用 `guided_selection + router_only` 组合，绕开识别和执行外呼

### 4.3 第一阶段建议

第一阶段不测真实自然语言识别，而测“已选意图驱动的路由图执行”：

- `guided_selection` 直接构造目标 graph
- `router_only` 避免执行下游 agent
- 通过结构化场景复用固定意图组合和槽位数据

这样可以稳定测量 router 核心开销，而不被模型和外部 agent 抖动污染。

## 5. 新增测试 Deployment

### 5.1 资源命名

新增以下资源：

- `Deployment/router-api-test`
- `Service/router-api-test`

### 5.2 部署要求

该 Deployment 与当前 `intent-router-api` 分离，配置独立环境变量：

- 启用压测挡板
- 指向与正式服务一致的 catalog 配置
- 保持独立的副本和资源配额，避免影响正式链路

### 5.3 访问方式

`admin-api` 调用压测目标时统一使用：

`http://router-api-test.intent.svc.cluster.local:8000/api/router`

这样可以避免以下干扰：

- 浏览器到集群入口的公网波动
- Ingress 转发链路差异
- 用户本地网络抖动

## 6. admin-api 压测能力设计

### 6.1 责任边界

`admin-api` 新增性能测试域能力，负责：

- 加载测试用例
- 校验压测参数
- 启动压测任务
- 并发执行请求
- 汇总阶段结果
- 输出总报告与失败样本

### 6.2 建议接口

建议新增以下接口：

- `GET /api/admin/perf-tests/cases`
- `POST /api/admin/perf-tests/runs`
- `GET /api/admin/perf-tests/runs/{run_id}`
- `GET /api/admin/perf-tests/runs/{run_id}/events`
- `GET /api/admin/perf-tests/runs`

### 6.3 任务模型

每个压测任务建议包含：

- `run_id`
- `case_id`
- `target_base_url`
- `ladder_steps`
- `status`
- `started_at`
- `finished_at`
- `aggregate_metrics`
- `step_results`
- `error_samples`

### 6.4 阶梯压测模型

每个阶梯步骤建议包含：

- 并发数
- 持续时长
- 预热时长
- 请求总量上限
- 每步间隔

示例：

- 10 并发，60 秒
- 30 并发，60 秒
- 50 并发，60 秒
- 100 并发，60 秒
- 200 并发，60 秒
- 300 并发，60 秒

### 6.5 指标

每步和全局都应统计：

- 总请求数
- 成功数
- 失败数
- 成功率
- 吞吐量（RPS）
- 平均耗时
- P50
- P95
- P99
- 最大耗时
- 超时数
- HTTP 状态码分布
- 错误类型分布

## 7. 测试用例设计

### 7.1 设计原则

- 测试用例使用结构化配置，不写死在前端页面
- 同一测试用例可重复执行
- 支持单场景和多场景组合

### 7.2 第一批建议用例

建议先提供 3 类场景：

#### 用例 A：单意图轻量场景

- 单个 `guided_selection` 节点
- 少量槽位
- 用于测基础请求处理和 session 开销

#### 用例 B：多意图顺序图场景

- 多个已选意图串联
- 模拟图编排与多节点状态流转
- 用于测 graph 构建和 drain 过程

#### 用例 C：槽位较多场景

- 单意图或双意图
- 提供较完整 slot memory
- 用于测上下文构建和结构化数据处理开销

### 7.3 请求模式

首期建议统一走以下链路：

1. 创建 session
2. 发送 `guided_selection` 消息
3. 使用 `router_only` 执行
4. 记录返回耗时与状态

如后续需要，再增加：

- 普通 `messages` 场景
- SSE streaming 场景

## 8. admin-web 性能测试页面设计

### 8.1 页面目标

该页面是运维/测试工作台，不是营销页。页面需要强调：

- 可配置
- 可观测
- 可回放
- 可对比

### 8.2 页面结构

建议新增独立页面，包含以下区域：

1. 测试配置区
- 选择测试用例
- 选择目标环境
- 配置阶梯步骤
- 配置每步持续时间、预热时间、超时阈值

2. 实时运行总览
- 当前阶段
- 当前并发
- 当前 RPS
- 当前成功率
- 当前 P95/P99
- 当前累计错误数

3. 阶梯结果表
- 每个步骤一行
- 展示并发、请求数、成功率、RPS、P50/P95/P99、错误数

4. 失败样本区
- 最近失败请求
- 错误信息
- 状态码
- 耗时
- 请求摘要

5. 历史记录区
- 最近运行记录
- 支持查看详情
- 支持对比不同运行结果

### 8.3 视觉与交互要求

页面风格应以专业控制台为目标：

- 低饱和冷色主基调
- 强层级的数字信息展示
- 重点指标高对比度突出
- 异常状态使用明显告警色
- 避免泛滥卡片堆砌
- 支持桌面和移动端可读性

交互上建议支持：

- 启动前参数校验
- 运行中进度刷新
- 结束后结果冻结
- 一键复制关键结果摘要

## 9. 验收标准

结合仓库现有文档中的压测目标，第一阶段建议验收线如下：

- 300 并发无报错
- P99 < 10s
- 阶梯过程可完整回放
- 可以识别失败样本与错误分布
- 压测全程不发生真实大模型 I/O

## 10. 实施顺序

建议按以下顺序推进：

1. 增加 `router-api` 挡板配置
2. 新建 `router-api-test` Deployment 与 Service
3. 在 `admin-api` 中实现压测任务执行与结果查询接口
4. 准备结构化测试用例
5. 在 `admin-web` 增加专业压测页面
6. 补充测试、部署说明与使用说明

## 11. 风险与注意事项

- 如果压测路径仍然走到真实识别或真实 planner，大模型挡板会导致失败，需要先约束测试输入路径
- 如果 `admin-api` 直接在请求线程内执行压测，可能阻塞普通管理接口，需要后台任务化
- 如果压测结果只保存在内存，Pod 重启后会丢失，后续可视需要增加持久化
- 如果 `router-api-test` 与正式服务共用资源配额，可能影响线上环境，需要资源隔离

## 12. 结论

本方案的核心是：

- 用 `router-api-test` 承载独立压测
- 用 `admin-api` 服务端发压
- 用集群内 `svc` 固定调用链路
- 用 LLM 挡板消除真实模型 I/O 干扰
- 用 `admin-web` 提供专业测试工作台

第一阶段优先验证 router 本身的吞吐、时延和稳定性；在结果稳定后，再决定是否扩展到真实 LLM 路径和真实 agent 路径的分层压测。
