v0.1 layered boundary guide

掌银助手、意图识别框架、子智能体的边界与落地路线

本文把系统拆成助手层、意图识别层、场景执行层，说明安全、限额、会话、意图识别、提槽、报文流转分别在哪里做，并给出 workflow / skill / API 的使用规范和典型场景推演。

总原则：Router 做路由，不执行业务；场景执行 Agent 做业务流程；掌银助手承载用户交互和上游上下文。

一、三层边界图

助手层 掌银助手前端 / 后端

面向用户的总控入口，负责承载会话、用户身份、页面上下文、展示交互、主动推送和最终话术。它不应该承载具体业务执行流程。

前端输入框、确认控件、页面上下文、结果展示。
助手后端统一会话入口、用户画像快照、渠道权限、可用 Agent 列表。
交互总控决定消息展示、卡片渲染、是否继续会话。
请求下行message + session_id + user/page context
↓
响应上行router decision / agent event / final response
意图识别层 Router / Intent Framework

负责识别用户要进入哪个场景，调用 spec 驱动识别器产出候选场景和路由槽位 hints，派发给对应执行 Agent，并追踪 Agent 任务状态。

识别场景召回、LLM 确认、歧义澄清。
提槽 hints只接收识别器按场景 spec 输出的槽位，并做白名单投影。
派发追踪创建 agent_task_id，记录 transcript 和 Agent event。
Agent Taskscene_id + raw_message + routing_slots + context refs
↓
Agent Eventwaiting_input / running / completed / failed
场景执行层 Sub Agents

由具体场景团队提供执行 Skill 和执行 Agent。业务提槽、确认、风控、限额、幂等、业务 API 调用、异常处理都在这里完成。

转账 Agent收款人/金额补槽、确认、风控、转账。
基金 Agent产品理解、风险揭示、申购试算、交易。
查询 Agent余额、账单、持仓等只读业务查询。

Spec 不是说明文档，而是运行时输入。平台定义格式、校验、加载和解释器；场景方提供具体 Router 路由 spec 和 Agent 执行 spec。

五、技术落地规范：workflow、skill、API 分别怎么用

类型
适合放什么
不适合放什么
在本体系里的位置
Workflow
确定性的状态流转、派发、回调追踪、业务步骤编排、异常分支。
大量业务说明文本、模糊知识、开放式推理。
Router 用 workflow 管路由生命周期；Agent 用 workflow 管业务执行生命周期。
Skill
场景说明、触发条件、槽位 schema、派发契约、业务执行说明、reference。
真实密钥、运行时状态、实时限额结果、最终交易事实。
Router 侧是路由 Skill；Agent 侧是执行 Skill。两者边界必须分开。
API
状态持久化、Spec Registry、Agent Dispatch、Agent Callback、业务后端。
让 LLM 直接自由选择任意 endpoint。
Router 只调用基础设施和 Agent API；业务 API 由执行 Agent 调用。

六、典型场景推演

用户输入

用户在掌银助手输入“给张三转5000块”。助手后端附带 `session_id`、用户画像快照、页面上下文和可派发 Agent 列表。

Router 加载路由上下文

读取 router session。如果没有未完成任务，则加载场景 index；如果已有 `agent_task_id`，优先恢复并转发到已有 Agent。

场景识别

LLM recognizer 用场景 metadata、routing spec 和 Skill metadata 判断是 `transfer` 场景。歧义时 Router 只做路由澄清，例如“是转账还是查转账记录”。

渐进读取路由 Skill

选中 `transfer` 后才读取场景提供的路由 spec：目标 Agent 是 `transfer-agent`，允许传递的 routing slot hints 是 `recipient` 和 `amount`。

识别器返回候选槽位

LLM recognizer 基于 routing spec 返回 `recipient=张三`、`amount=5000` 等 hints；Router runtime 只做白名单投影。这些只作为 `routing_slots`，执行 Agent 仍要做业务校验和必要补槽。

派发 Agent Task

Router 创建 `agent_task_id=task_001`，把 raw message、routing slots、profile/page context refs 派发给 `transfer-agent`。

Agent 执行业务流程

`transfer-agent` 处理业务提槽、风控、限额、确认、转账 API 和异常。Router 不参与这些业务步骤。

Router 追踪和回传

Agent 返回 waiting/running/completed 事件，Router 更新 transcript 和 routing state，再把消息交给助手层展示。

八、建议落地原则

先边界，后智能

先确定 Router 不能做什么，再接 LLM。边界不清时，模型能力越强越容易越权。

先状态，后压缩

多轮能不能接上，靠结构化 state，不靠摘要。摘要只是给模型看的辅助材料。

先派发，后执行

Router 的成功标准是正确派给场景 Agent，并追踪任务，不是替业务系统完成交易。
