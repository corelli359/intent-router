Router V4 / v0.2

架构方案与业务旅程

v0.2 在 v0.1 的 spec 驱动路由基础上，增加主动推送、任务图、Agent structured output、handover 兜底改派和任务级拆流预留。核心边界仍然不变：Router 做判断、计划、派发和追踪；业务执行和安全确认仍在场景 Agent；最终用户话术由助手生成。

一、v0.2 总体架构

助手层掌银助手前后端

入口

接收用户输入、页面上下文、用户画像和推荐卡片上下文。

展示

消费 Router 结构化状态、task/graph 进度和 Agent output。

最终表达

任务结束后基于 agent_output.data 生成用户可见结果。

Router 层router-v4-service

识别与计划

按 routing spec 识别场景；主动推送时受 push_context.intents 约束；多意图生成 graph/tasks。

派发与追踪

创建 task，派发执行 Agent，维护 session、task、graph、transcript 和 snapshots。

兜底改派

仅在 ishandover=true 且 output.data=[] 时改派 fallback-agent。

执行层场景 Agent

业务执行

业务提槽、确认、风控、限额、幂等和业务 API 调用。

结构化结果

只返回结构化状态和 output，不控制最终用户话术。

误派判断

发现不属于自己时按固定 handover 协议通知 Router。

Router 允许调用的 API 类型是基础设施 API：spec registry、session/task/graph store、Agent dispatch、Agent output callback。Router 不调用转账、申购、风控、限额等业务 API。

四、旅程 A：普通用户输入

用户输入

用户输入“给张三转5000块”，助手传入 session、用户画像和页面上下文。

加载路由上下文

Router 读取 session/transcript；若已有 active task，则直接转发给对应 Agent，不重新识别。

识别场景

根据 scene index 和 routing spec metadata 选中 transfer。

提取 routing slots

提取可直取 hints，例如 recipient=张三、amount=5000。

派发 Agent

创建 task，派发给 transfer-agent；Router 只追踪状态，不做业务确认、风控或交易 API。

助手展示

助手展示 Router/Agent 结构化状态；最终结果由助手基于 Agent output 组织。

五、旅程 B：主动推送推荐

助手推送卡片

助手直接推送推荐卡片，并在后续请求里携带 push_context.intents。

用户表达

用户说“就按这个办”。Router 不再问“是否接受推荐”。

候选约束

Router 优先在推荐意图清单内判断；泛化接受默认选择第一推荐项或卡片绑定项。

执行派发

Router 直接派发对应 Agent。业务安全确认仍由 Agent/业务域处理。

拒绝分支

用户说“不用了”时返回 no_action，不创建 task。

六、旅程 C：多意图拆流

用户表达多意图

例如“都看一下”，且 push context 里有余额和基金两个候选。

Router 创建 Graph

返回 status=planned、graph_id、stream_mode=split_by_task。

生成多个 Task

每个 task 包含 scene、target_agent、stream_url、resume_token。

助手按 Task 消费

v0.2 先提供 task snapshot 和 stream URL；真正 SSE resume 后续接入。

汇总结果

多个 Agent output 都完成后，助手汇总生成最终展示。

助手兜底展示结构化结果生成话术

触发条件

{
  "ishandover": true,
  "output": {"data": []}
}

不触发条件

{
  "isHandover": true,
  "output": {"data": []}
}

防循环规则：同一个原 task 最多 handover 一次；fallback-agent 如果再次返回 handover，Router 标记 handover_exhausted，不继续改派。
