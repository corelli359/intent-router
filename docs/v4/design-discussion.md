# 掌银智能体系统设计方案

## 1. 要解决什么问题

掌银 APP 需要一个 AI 助手，用户说"给张三转5000块"，系统能理解意图、收集参数、调后端接口完成转账。

核心挑战：怎么让 LLM **可控地**执行业务流程，而不是自由发挥。

我们的解法：**总控 + Skill**。LLM 负责理解和调度，业务流程写在 Skill（markdown 文件）里，LLM 严格按 Skill 定义的步骤执行。

## 2. 整体架构

```
┌──────────────┐
│   掌银 APP   │
└──────┬───────┘
       │ 请求参数：
       │   message: "给张三转5000块"

       ▼
┌─────────────────────────────────────────────────┐
│                   总 控                          │
│                                                  │
│  ┌──────────────────────────────────────────┐   │
│  │         System Prompt（自动构建）          │   │
│  │                                          │   │
│  │  ① agent.md    — 框架执行规则             │   │
│  │  ② 入参注入    — 用户画像、页面上下文      │   │
│  │  ③ 短期记忆    — 当前对话历史              │   │
│  │  ④ Skill索引   — 所有Skill名称+摘要       │   │
│  └──────────────────┬───────────────────────┘   │
│                     │                            │
│                     ▼                            │
│  ┌──────────────────────────────────────────┐   │
│  │              LLM (Qwen)                   │   │

│  └──────┬───────┬────────┬──────────┬───────┘   │
│         │       │        │          │            │
│    skill_read  rag_query api_call  memory_query  │
│         │       │        │          │            │

│    ┌────────┐ ┌────┐ ┌──────┐ ┌────────┐       │
│    │skills/ │ │知识│ │后端API│ │ 长期    │       │
│    │*.md    │ │库  │ │      │ │ 记忆    │       │
│    └────────┘ └────┘ └──────┘ └────────┘       │
└─────────────────────────────────────────────────┘
```

**关键点**：用户画像、业务API地址、页面上下文都是调用方传进来的参数，总控不自己去拉取。

## 3. 两个核心文件

### 3.1 agent.md — 总控的行为规则

agent.md 是总控的"宪法"，定义 LLM 怎么工作。所有执行逻辑都在这个文件里描述，代码不硬编码流程。

```
agent.md 做什么：
  ✅ 定义角色（你是一个调度器，不直接处理业务）
  ✅ 定义上下文加载时序（什么时候加载什么）
  ✅ 定义工具使用规则（什么时候调 rag_query，什么时候调 skill_read）
  ✅ 定义边界（不能编造数据，不能跳过步骤）

agent.md 不做什么：
  ❌ 不定义具体业务流程（那是 Skill 的事）
  ❌ 不定义具体参数（那是 Skill 的事）
```

### 3.2 skills/*.md — 业务流程定义

每个 Skill 是一个 markdown 文件，由业务团队编写。总控读完后按步骤执行。

```markdown
# 转账 Skill

## 触发条件
转账、打款、汇款、打钱、给某人转钱

## 参数
| 参数 | 必填 | 说明 |
|---|---|---|
| recipient | 是 | 收款人 |
| amount | 是 | 转账金额 |

## 业务流程
### 步骤1：信息补全
缺参数时追问，一次只问一个。

### 步骤2：风控检查
调用 api_call("/api/risk/check", ...)

### 步骤3：用户确认
展示摘要，等用户说"确认"才继续。

### 步骤4：执行转账
调用 api_call("/api/transfer", ...)

### 步骤5：反馈结果

## 异常处理
余额不足 → 告知余额，建议换金额
超限 → 告知限额

## References
- references/risk_rules.md
- references/transfer_limits.md
```

**新增一个 Skill = 写一个 .md 文件**，不需要改代码。

## 4. 上下文加载时序

上下文不是一次性全塞给 LLM，而是分阶段按需加载。这直接控制 token 消耗和响应质量。

```
┌─────────────────────────────────────────────────────────────┐
│ 阶段A：请求进来，代码自动构建 System Prompt                    │
│                                                              │
│   agent.md（全文）           ← 框架规则                       │
│   + 入参.user_profile        ← 调用方传入的用户画像            │
│   + 入参.page_context        ← 调用方传入的页面上下文          │
│   + session.history          ← 短期记忆（对话历史）            │
│   + skill索引（名称+摘要）    ← 扫描 skills/*.md 生成          │
│                                                              │
│   → 拼成一个完整的 System Prompt，发给 LLM                    │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段B：LLM 意图识别（内部推理，不调工具）                      │
│                                                              │
│   三种结果：                                                  │
│   ① 匹配到 Skill → 进入阶段C                                │
│   ② 用户问了专业术语 → rag_query → 回答 → 结束               │
│   ③ 意图模糊 → 追问用户 → 等回复 → 回到B                     │
└──────────────────────────┬──────────────────────────────────┘
                           ▼ (仅①)
┌─────────────────────────────────────────────────────────────┐
│ 阶段C：LLM 按需加载 Skill 上下文                              │
│                                                              │
│   C1  skill_read("transfer")    ★必须：读取Skill完整内容      │
│   C2  memory_query("转账偏好")  ○按需：查长期记忆              │
│   C3  rag_query("什么是QDII")   ○按需：仅当用户输入有专业术语   │
│   C4  skill_read("references/风险规则.md")  ○按需：Skill引用时 │
│                                                              │
│   注意：C3 只看用户输入，Skill 内部不清楚的术语走 C4           │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段D：参数提取                                               │
│                                                              │
│   从已有上下文提取 Skill 需要的参数                            │
│   缺参数 → 追问用户 → 回到D                                   │
│   参数完整 → 进入E                                            │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段E：按 Skill 步骤执行                                     │
│                                                              │
│   严格按 Skill 定义的步骤顺序调用工具                          │
│   每一步按需调 api_call / 等用户确认 / 展示结果                │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段F：反馈结果                                               │
│                                                              │
│   用自然语言反馈给用户，记录到对话历史                          │
└─────────────────────────────────────────────────────────────┘
```

### RAG vs Reference 的分工

```
                    用户输入 "我想买个QDII基金"
                              │
                    ┌─────────┴──────────┐
                    │ "QDII"是专业术语吗？ │
                    └─────────┬──────────┘
                         是 ↓        否 → 不调
                    ┌───────────┐
                    │ rag_query │ ← 查知识库，理解用户在说什么
                    └───────────┘


                    Skill 内部提到"R3风险等级"
                              │
                    ┌─────────┴──────────┐
                    │  Skill 有 reference  │
                    │  章节吗？            │
                    └─────────┬──────────┘
                         有 ↓        否 → 不调
                    ┌──────────────┐
                    │ skill_read   │ ← 读 references/risk_level.md
                    │ (reference)  │    理解 Skill 内部术语
                    └──────────────┘
```

一句话：**RAG 管用户端的理解，Reference 管 Skill 端的理解。**

## 5. 一次完整交互的全过程

**场景**：用户第一次说"帮我转账"，然后被追问，最终完成。

```
用户: "帮我转账"
  │
  ├─ 阶段A: 代码构建 System Prompt
  │    [agent.md] + [用户画像: I类账户, 稳健型] + [对话历史: 空] + [Skill索引]
  │
  ├─ 阶段B: LLM → "转账" → 匹配到 transfer Skill
  │
  ├─ 阶段C:
  │    C1: skill_read("transfer") → 获得5步流程
  │    C2: 跳过
  │    C3: "帮我转账" 没有专业术语 → 跳过
  │    C4: transfer.md 引用了 risk_rules.md → skill_read("references/risk_rules.md")
  │
  ├─ 阶段D: 提取参数 → recipient=缺失, amount=缺失
  │    → 追问: "请问您要转给谁？"
  │
  ▼
用户: "张三"
  │
  ├─ 阶段D: 提取 → recipient="张三", amount=缺失
  │    → 追问: "转多少钱？"
  │
  ▼
用户: "5000"
  │
  ├─ 阶段D: 提取 → recipient="张三", amount="5000" → 完整
  │
  ├─ 阶段E: 按 transfer.md 步骤执行
  │    步骤2: api_call("/api/risk/check", {recipient:"张三", amount:"5000"}) → 通过
  │    步骤3: 展示 "确认转账5000元给张三？" → 等用户确认
  │
  ▼
用户: "确认"
  │
  ├─ 阶段E (续):
  │    步骤4: api_call("/api/transfer", {recipient:"张三", amount:"5000"}) → 成功
  │
  ├─ 阶段F: "已成功转账5000元给张三，交易流水号：TXN001"
  │
  ▼
结束
```

## 6. 调用接口设计

掌银后端调用总控的接口：

### 请求

```json
{
  "session_id": "sess_abc123",
  "message": "给张三转5000块",
  "user_profile": {
    "user_id": "U001",
    "account_type": "I类",
    "risk_level": "C3",
    "frequent_contacts": ["张三", "李四"],
    "available_balance": 50000
  },
  "page_context": {
    "current_page": "首页",
    "selected_item": null
  },
  "business_apis": {
    "transfer": "/api/transfer",
    "risk_check": "/api/risk/check",
    "fund_detail": "/api/fund/detail",
    "fund_holdings": "/api/fund/holdings",
    "recommend": "/api/recommend"
  }
}
```

### 响应

```json
{
  "session_id": "sess_abc123",
  "response": "确认转账5000元给张三？",
  "action_required": {
    "type": "confirm",
    "data": {
      "recipient": "张三",
      "amount": 5000
    }
  },
  "skill": "transfer",
  "skill_step": 3,
  "tool_calls_log": [
    {"tool": "skill_read", "args": {"name": "transfer"}},
    {"tool": "skill_read", "args": {"name": "references/risk_rules.md"}},
    {"tool": "api_call", "args": {"endpoint": "/api/risk/check", "body": {"recipient":"张三","amount":"5000"}}}
  ]
}
```

**所有外部依赖通过入参传入**，总控自己不拉取用户画像、不硬编码业务API地址。

**新增 Skill = 在 skills/ 目录下新建一个 .md 文件**。不需要改代码、不需要重启服务（下次请求自动扫描）。

## 9. 上下文管理：压缩与卸载

对话历史会不断增长，Skill 内容可能占几千 token，不做管理很快就超限。

### 9.1 Skill 的加载-使用-卸载生命周期

```
    skill_read("transfer")
           │
           ▼
    ┌──────────────┐
    │  Skill 已加载  │  ← ToolMessage 进入对话历史
    │  阶段C: 读取   │
    │  阶段D: 提参数  │
    │  阶段E: 执行   │
    └──────┬───────┘
           │
           │ Skill 执行结束（成功/失败/取消）
           ▼
    ┌──────────────┐
    │  Skill 卸载   │  ← 从对话历史中移除 Skill 的 ToolMessage
    │               │     保留执行结果的摘要
    └──────────────┘
```

**卸载规则**：

| 事件 | 动作 | 原因 |
|---|---|---|
| Skill 执行成功 | 移除 Skill 内容 + reference，保留结果摘要（"已转账5000元给张三"） | Skill 内容可重新加载，结果需要保留 |
| Skill 执行失败 | 移除 Skill 内容，保留失败原因摘要 | 后续可能换 Skill 重试 |
| 用户取消 Skill | 移除全部 Skill 相关内容 | 用户主动放弃，不需要保留 |
| 用户切换话题 | 先确认放弃当前 Skill → 确认后卸载 | 避免浪费上下文空间 |

**卸载后如果需要重来**：LLM 重新 skill_read 加载即可，内容不会丢。

### 9.2 对话历史压缩

不是等到快爆了才压缩，而是每轮对话结束后主动检查。

```
每轮对话结束后：
    │
    ├─ 检查1: Skill 是否已完成？ → 是 → 卸载（见上表）
    │
    ├─ 检查2: 追问轮次 > 3？ → 是 → 压缩追问过程
    │   │
    │   │  压缩前（4条消息）：
    │   │    User: "帮我转账"
    │   │    AI: "转给谁？"
    │   │    User: "张三"
    │   │    AI: "转多少？"
    │   │
    │   │  压缩后（1条摘要）：
    │   │    [上下文摘要: 用户要转账给张三，金额待确认]
    │
    ├─ 检查3: 对话总轮次 > 阈值（如10轮）？ → 是 → 压缩早期对话
    │   │
    │   │  保留最近3轮原文
    │   │  更早的对话压缩为摘要：
    │   │    [已完成操作: 转账5000元给张三(TXN001)]
    │   │    [当前意图: 用户在问基金推荐]
    │   │    [已提取参数: 风险等级C3]
    │
    └─ 检查4: 总 token 数 > 模型上限的70%？ → 是 → 强制压缩
        │
        │  按优先级丢弃（从最不重要的开始）：
        │  1. 已卸载 Skill 的残余 ToolMessage
        │  2. api_call 的完整响应（保留关键结果字段）
        │  3. reference 内容（可重新加载）
        │  4. 早期追问过程（保留最终参数）
        │  最后才压缩：用户的原始消息（最有价值）
```

### 9.3 压缩后保留什么

```
压缩后的对话历史结构：

┌─────────────────────────────────────┐
│ [摘要区]                             │
│  已完成操作: 转账5000元给张三(TXN001) │
│  当前意图: 用户在问基金              │
│  已提取参数: risk_level=C3           │
├─────────────────────────────────────┤
│ [最近3轮原文]                        │
│  User: 有什么基金推荐吗              │
│  AI: [推荐结果]                      │
│  User: 沪深300ETF怎么样              │
├─────────────────────────────────────┤
│ [当前Skill区]（如果正在执行）         │
│  Skill: fund_query                   │
│  步骤: 3                             │
│  已提取参数: fund_name=沪深300ETF    │
└─────────────────────────────────────┘
```

**关键原则**：
- **可重新获取的内容优先丢弃**（Skill 内容、reference、api 响应）
- **不可恢复的内容优先保留**（用户的原始表达、操作结果、当前状态）
- 压缩由代码执行，不是让 LLM 自己压缩（避免 LLM 丢失关键信息）

## 10. 需要讨论的问题

| # | 问题 | 选项 | 建议 |
|---|---|---|---|
| 1 | **Skill 匹配方式**：纯 LLM 看索引匹配，还是 embedding 向量匹配先筛一轮？ | A. 纯LLM匹配 / B. embedding预筛+LLM确认 | B（Skill多时纯LLM不稳定） |
| 2 | **长期记忆存哪里**：本地SQLite、Redis、还是调用现有记忆服务？ | A. SQLite / B. Redis / C. 外部服务 | 看现有基础设施 |
| 3 | **Skill 步骤执行失败怎么办**：LLM 自主处理，还是返回错误让调用方决定？ | A. LLM按Skill异常处理规则处理 / B. 透传错误给调用方 | A（Skill已定义异常处理） |
| 4 | **并发会话**：多个用户同时使用，Session 隔离怎么做？ | A. 进程内dict / B. Redis / C. 外部存储 | 看部署方式 |
