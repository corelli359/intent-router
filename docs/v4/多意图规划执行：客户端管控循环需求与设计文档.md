# 多意图规划执行：客户端管控循环需求与设计文档

> 状态：草案 v0.1（待评审）
> 适用分支：`fix/subagent-call`
> 作者：router-service
> 目标读者：router / assistant / 客户端同事

---

## 1. 背景

### 1.1 现状

当前分支已经实现了完整的图规划 + 图运行时：

- 多意图识别（`recognizer`）+ LLM 图规划（`LLMIntentGraphPlanner`）→ `ExecutionGraphState` 多节点图
- 节点级 `WAITING_USER_INPUT / WAITING_CONFIRMATION / WAITING_ASSISTANT_COMPLETION` 状态
- 客户端显式完成信号 `POST /api/v1/task/completion`（commit `e99a785` / `ea9a834`）
- 中途换意图 → `TurnInterpreter.interpret_waiting_node` 返回 `replan` → 整图替换
- 每次响应携带 `task_list`，客户端总能看到最新任务列表

### 1.2 问题

默认 `_drain_graph` 是**服务端自动串行推进**：一个节点完成后立刻拿 `next_ready_node` 派下一个，只有碰到 `WAITING_*` 或全部终结才返回。

业务侧希望**客户端管理任务循环**：
- 每次请求最多推进一个节点；
- 节点做完后由客户端显式决定"开始下一个"；
- 任务之间客户端可以插入自己的交互（总结、二次确认、跨会话动作等）。

### 1.3 目标

在现有实现上加一个"单节点推进"执行模式，不重写既有组件。协议层面：

- 新增 `executionMode=per_node` 档位；
- 复用已有 `POST /api/v1/task/completion` 作为唯一完成通道；
- 复用已有 `node.waiting_user_input` 事件作为反问统一载体；
- 复用已有 replan 路径实现换意图整图替换。

## 2. 术语

| 术语 | 含义 |
|---|---|
| 意图 (Intent) | 注册在 catalog 里的业务类型，如 `transfer_money`、`gas_bill` |
| 节点 (Node) | 执行图上的一个意图实例，`GraphNodeState` |
| 图 (Graph) | 一轮规划产物，`ExecutionGraphState`，由 nodes + edges 构成 |
| 任务 (Task) | 节点就绪后分派给 sub-agent 的执行单元，`Task` |
| 反问 (Reask) | 系统要求用户补充信息，可能来自 router 槽位校验或 sub-agent 业务侧 |
| 槽位 (Slot) | 意图执行所需的结构化参数 |
| 完成信号 (Completion Signal) | `POST /task/completion` 带 `completionSignal ∈ {1,2}`，2 表示完全结束 |
| drain | 编排器循环派发就绪节点的过程 |
| replan | 识别到新意图时整图重建 |

## 3. 决策汇总（已拍板）

| 维度 | 决策 | 理由 |
|---|---|---|
| 任务推进模式 | **客户端单节点推进** | 客户端需要在节点之间插入自己的交互；服务端自动 drain 会剥夺控制权 |
| 完成判定 | **客户端显式发 `POST /task/completion`** | 业务侧 UI 决定何时结束，sub-agent 自报 COMPLETED 仅作信息事件 |
| 槽位提取层级 | **router 先提取 + 硬门** | 复杂度集中在 router，sub-agent 实现更薄；现有 validator 已完备 |
| 换意图行为 | **整图替换** | 协议最简单，前端只看 task_list 新版本；旧图 SUSPENDED 保留审计 |

## 4. 需求场景

### 4.1 典型场景：多意图 + 全流程

**对话**："先给张三转1000，再交3月份水费，户号12345"

| 轮次 | 客户端请求 | 服务端响应要点 | 图状态 |
|---|---|---|---|
| T1 | `POST /message` txt="先给张三转1000，再交3月份水费，户号12345" mode=per_node | 识别 2 意图 → 规划图 → `graph.proposed`（WAITING_CONFIRMATION）+ task_list[转账 waiting, 缴费 waiting] | WAITING_CONFIRMATION |
| T2 | `POST /action` code=confirm_graph | 激活图 → 节点1（转账）：router 提槽 ✓ → 分派 agent → agent 流输出 → 节点=RUNNING | RUNNING（仅推进节点1） |
| T3 | `POST /task/completion` taskId=节点1 signal=2 | 节点1=COMPLETED；**per_node 模式下不自动推进节点2** → task_list[转账 completed, 缴费 waiting] | RUNNING |
| T4 | `POST /message` txt="继续" 或 txt="" | 无 waiting 节点 → 推进节点2 → router 提槽：月份=3、户号=12345 ✓ → 分派 agent → 节点=RUNNING | RUNNING（仅推进节点2） |
| T5 | `POST /task/completion` taskId=节点2 signal=2 | 节点2=COMPLETED；图=COMPLETED → task_list[all completed] | COMPLETED |

### 4.2 分支：router 层反问（槽位不足）

T2 分支：用户只说了"给张三转钱"，没说金额
- `_validate_node_understanding` 发现 `amount` 缺失 → `_mark_node_waiting_for_slots`
- 节点 → WAITING_USER_INPUT，图 → WAITING_USER_INPUT
- 响应 output 带反问文本"请问转账金额？"，不分派 agent
- T2' 客户端 `POST /message` txt="1000"
- `TurnInterpreter.interpret_waiting_node` → `resume_current`
- `_resume_waiting_node` → 重新校验槽位 ✓ → 分派 agent → 节点=RUNNING

### 4.3 分支：agent 层反问

- sub-agent 流中返回 `status=WAITING_USER_INPUT` + 反问文本
- `_handle_agent_chunk` 把节点置为 WAITING_USER_INPUT，图同步
- 客户端的反问载体**与 router 反问一致**（`node.waiting_user_input`）
- 下一轮 `POST /message` 补输入 → resume_current → agent_client.stream 重新调用

### 4.4 分支：中途换意图

- 任意 T 轮用户说："不转了，查下我的余额"
- 入口：`_route_new_message` 或 `interpret_waiting_node` 检测到新主意图 → `replan`
- 服务端：
  - 旧图标记 SUSPENDED（保留在 session 供审计）
  - 新建图（仅"查余额"一个节点）
  - 响应 `graph.proposed` + 新 task_list
- 客户端 UI 整体刷新任务列表，继续 per_node 循环

### 4.5 分支：取消当前节点 / 整图

- 节点级取消：`POST /action` code=cancel_node nodeId=xxx → 节点=CANCELLED → 下游按依赖传播为 SKIPPED
- 整图取消：`POST /action` code=cancel_graph → 图=CANCELLED → session 进入空闲

### 4.6 分支：单意图（退化单节点图）

- 用户说"查余额" → 识别 1 意图 → 图=DRAFT（无需确认）→ 直接推进
- per_node 模式下，T1 响应已经推进了节点 → T2 客户端发 completion 即可结束

### 4.7 分支：图确认被拒

- T2' 客户端发 `POST /action` code=cancel_graph（拒绝执行）
- 图=CANCELLED，session 回到空闲态

## 5. 非功能需求

| 维度 | 要求 |
|---|---|
| 幂等 | 同一 taskId 的 completion 可重发；重复的 confirm_graph 无副作用 |
| 并发 | 同一 session 上 completion 与 message 并发到达时，依赖现有 `*_serialized` 串行化 |
| 超时 | sub-agent 流超时按现有策略（agent_client 层）；不引入新超时 |
| 观测 | 保留现有 diagnostics / 日志；新增 per_node 模式标签便于区分统计 |
| 版本化 | `task_list` 隐含 graph_version，替换图时版本号变化，客户端据此决定是否整体刷新 |
| 兼容 | `executionMode=execute`（现状）继续支持，不影响已上线客户端 |

## 6. 协议规范

### 6.1 请求

#### `POST /api/v1/message`（非流）/ `/api/v1/message/stream`（流）

```json
{
  "sessionId": "sess_xxx",
  "txt": "先转账再缴费",
  "stream": false,
  "executionMode": "per_node",        // 新增取值：execute | router_only | per_node
  "config_variables": {...}
}
```

#### `POST /api/v1/action`

```json
{
  "sessionId": "sess_xxx",
  "actionCode": "confirm_graph | cancel_graph | cancel_node",
  "nodeId": "..."                     // 仅 cancel_node 需要
}
```

#### `POST /api/v1/task/completion`

```json
{
  "sessionId": "sess_xxx",
  "taskId": "task_xxx",
  "completionSignal": 2               // 1=partial 2=fully_done
}
```

**约定**：completion 接口本身**不触发** drain 继续分派下一个节点；仅修改节点状态并回显最新 task_list。

### 6.2 响应

#### 非流式统一信封 `AssistantOutputResponse`

```json
{
  "ok": true,
  "output": {
    "text": "...",                    // 反问文本或 agent 输出
    "payload": {...}                  // 结构化数据
  },
  "task_list": [
    {"taskId": "...", "name": "转账#0", "status": "running"},
    {"taskId": "...", "name": "缴费#1", "status": "waiting"}
  ],
  "graphStatus": "running"
}
```

#### 流式 SSE（白名单事件，通过 `event: message` 投递）

| 事件 | 触发时机 | payload 关键字段 |
|---|---|---|
| `graph.unrecognized` | 识别不到有效意图 | reason |
| `graph.proposed` | 新图生成等待确认 | graph, task_list |
| `graph.waiting_confirmation` | 确认阶段超时/重发 | task_list |
| `node.ready_for_dispatch` | 节点进入 READY | node |
| `node.message` | agent 流式输出 | node, chunk |
| `node.waiting_user_input` | router 或 agent 反问 | node, prompt |
| `node.waiting_confirmation` | 节点需要确认 | node |
| `node.waiting_assistant_completion` | 等待客户端发 completion | node |
| `node.completed` / `node.failed` / `node.cancelled` | 节点终结 | node, task_list |

终止帧：`event: done`

### 6.3 `task_list` 字段

- `taskId`：与 completion 接口的 taskId 对齐
- `name`：`intent_code#position` 或显示名
- `status`：`waiting | running | completed | failed | cancelled | skipped`
- 顺序：按 `node.position`，反映图 topology 而非实时调度顺序

### 6.4 `completionSignal` 语义

| 值 | 含义 |
|---|---|
| 1 | 部分完成（agent_partial_done），仅用于统计，不改变节点状态 |
| 2 | 完全完成，节点 → COMPLETED，图状态刷新 |

## 7. 状态机

### 7.1 节点状态

```
PENDING → BLOCKED → READY → DISPATCHING → RUNNING → COMPLETED
                              ↓                ↓
                     WAITING_USER_INPUT ← ─ ─ ┘
                     WAITING_CONFIRMATION
                     WAITING_ASSISTANT_COMPLETION
                              ↓
                           RUNNING（resume 后）
                              ↓
                          CANCELLED / FAILED / SKIPPED
```

- `BLOCKED → READY`：所有上游终结且条件满足
- `BLOCKED → SKIPPED`：上游 FAILED / CANCELLED / SKIPPED 或条件不满足
- `RUNNING → WAITING_*`：router 槽位校验失败 或 agent 主动挂起
- `RUNNING → COMPLETED`：**仅由客户端 `/task/completion` signal=2 驱动**（决策锁定）
- `COMPLETED → ...`：终态

### 7.2 图状态

```
DRAFT / WAITING_CONFIRMATION → RUNNING → COMPLETED
                                   ↓
                              WAITING_USER_INPUT（任一节点 waiting）
                                   ↓
                                RUNNING
                                   ↓
                            CANCELLED / FAILED / SUSPENDED（被 replan 替换时）
```

### 7.3 `per_node` 模式下 drain 的停止条件（新）

Drain 循环在以下任一条件返回：
1. 所有节点终结（图 COMPLETED / FAILED / CANCELLED）；
2. 存在 `WAITING_*` 节点（沿用现状）；
3. **刚完成一个节点（状态转为 COMPLETED）→ 立即返回，即使还有 READY 节点**（新增）；
4. 达到 `max_drain_iterations` 兜底。

## 8. 核心时序

### 8.1 多意图全流程（对照 §4.1）

```
Client                       Router                 SubAgent
  │  message("...转账...缴费") │
  ├───────────────────────────>│
  │                            │ recognize → plan(2 nodes)
  │  graph.proposed + list     │
  │<───────────────────────────┤
  │  action(confirm_graph)     │
  ├───────────────────────────>│
  │                            │ activate → node1 READY → 槽位 ✓
  │                            │ dispatch ──────────────────> agent1
  │                            │                              stream chunks
  │  node.message (agent1)     │<──────────────────────────── 
  │<───────────────────────────┤
  │  [per_node return 节点1 RUNNING]
  │  task/completion(node1)    │
  ├───────────────────────────>│ node1 → COMPLETED; 不 drain 下一节点
  │  list[node1=done]          │
  │<───────────────────────────┤
  │  message("继续")            │
  ├───────────────────────────>│ 无 waiting → 推进 node2 → 槽位 ✓
  │                            │ dispatch ──────────────────> agent2
  │                            │                              stream chunks
  │  node.message (agent2)     │<────────────────────────────
  │<───────────────────────────┤
  │  task/completion(node2)    │
  ├───────────────────────────>│ node2 → COMPLETED; graph COMPLETED
  │  list[all done]            │
  │<───────────────────────────┤
```

### 8.2 中途换意图（对照 §4.4）

```
Client                        Router
  │  [已在执行节点1，某轮返回后]
  │  message("不转了，查余额")   │
  ├─────────────────────────────>│ turn_interpreter → replan
  │                              │ 旧图 → SUSPENDED
  │                              │ 识别 → 规划 → 新图（单节点）
  │  graph.proposed + new list   │
  │<─────────────────────────────┤
  │  [继续 per_node 循环...]
```

### 9.1 改动范围

| `docs/v3/archive/router-service-助手对接接口文档-v0.4.md` | 新增执行模式章节 + 客户端驱动循环时序 |
| `backend/services/router-service/tests/` | 新增 `test_per_node_drain.py` |

### 9.2 drain 策略伪代码

```python
async def _drain_graph(session, seed_input, *, execution_mode):
    iterations = 0
    max_iter = self._resolve_max_drain_iterations(graph)
    while True:
        iterations += 1
        if iterations > max_iter:
            return await self._fail_drain_graph(...)

        await self._refresh_graph_state(session, graph)
        if waiting_node := self._get_waiting_node(session):
            return   # 既有行为

        next_node = self._next_ready_node(graph)
        if next_node is None:
            return   # 全部终结

        before_status = next_node.status
        await self._run_node(session, graph, next_node, seed_input)
        # per_node 新增：节点刚完成（非 WAITING_*）立即返回
        if execution_mode == "per_node" and next_node.status == GraphNodeStatus.COMPLETED:
            return
```

注意：`_run_node` 返回后节点可能是 `COMPLETED / WAITING_* / FAILED`。WAITING_* 会在下一轮循环 `_get_waiting_node` 捕获后返回；COMPLETED 在 per_node 模式下新增提前 return；FAILED 沿用现有失败传播。

### 9.3 配置透传路径

```
HTTP request (executionMode=per_node)
  → sessions.py: ProtocolMessageRequest
  → orchestrator.handle_user_message_serialized(..., execution_mode=...)
  → message_flow.route_new_message(..., execution_mode=...)
  → orchestrator._drain_graph(..., execution_mode=...)
```

不放 session 级配置，避免跨请求粘滞；以请求参数为准，每个请求独立。

## 12. 兼容与迁移

- 默认 `executionMode=execute` 保持原语义，已上线客户端无需改动。
- assistant-service 升级时可通过 feature flag 选择 per_node，灰度推进。
- `task/completion` 接口早已存在，per_node 下使用方式不变，只是含义更明确（它不再是 agent 自动完成的补丁，而是**唯一完成通道**）。

## 13. 风险与回滚

| 风险 | 影响 | 缓解 |
|---|---|---|
| 客户端忘记发 completion → 任务永远卡 RUNNING | 会话无法推进 | 加超时兜底：节点 RUNNING 超过 N 分钟告警；提供 `force_complete` 运维接口 |
| per_node 下 task_list 频繁刷新前端 | 性能 | task_list 带版本号，前端只在变化时刷新 |
| replan 时客户端未及时感知 | 客户端以旧 taskId 发 completion | 服务端对未知 taskId 返回 404，前端回退重拉任务列表 |
| 旧图 SUSPENDED 累积 | session 膨胀 | 会话级最多保留最近 3 个 suspended 图，超出淘汰 |

回滚策略：`per_node` 走独立代码路径（新分支条件），出问题直接关功能开关，回退到 `execute` 模式不影响其他通路。

## 14. 开放问题

1. **空消息是否允许推进下一节点**：T4 客户端 `POST /message txt=""` 作为"继续"信号是否合理？或应使用 `POST /action code=advance`？
2. **completion 响应是否该内嵌"下一节点预览"**：让客户端少发一次 `/message`？权衡：协议更简单 vs. 响应体更重。
3. **multi-branch 图（并行）支持时机**：当前 drain 仍是串行；`per_node` 对并行图的语义需要再定义（一次推进 N 个？还是一次只推一条路径？）。暂定并行能力不在本期，后续单独评估。
4. **历史 SUSPENDED 图是否对客户端可见**：通过 snapshot 接口返回？还是独立 `/sessions/{id}/history` 接口？
