v0.1 architecture walkthrough

Spec 驱动运行时怎么搭建，怎么从一次用户输入跑完整流程

这份图解只讲两件事：代码层面如何搭建 spec 驱动能力；用户输入以后，系统如何渐进加载状态、Skill、Reference 和业务能力，并在多轮里继续执行。

最重要的边界：LLM 不直接执行业务。LLM 只参与理解和建议，Runtime 持有状态、校验步骤、控制 capability、写审计。

一、代码层面应该搭成什么样

助手层 掌银前端 / 后端

面向用户，承载会话入口、页面上下文、用户画像快照、展示与交互。它把用户输入和上下文传给 Router，并把 Router 或 Agent 的响应展示给用户。

前端 用户输入、页面状态、确认控件、结果展示。
助手后端 session_id、用户画像、渠道权限、可派发 Agent 信息。
上游事实 user_profile、page_context、入口来源、用户原文。
请求下行 message + session_id + context
↓
响应上行 response + action/status
意图识别层 Router

做路由，不执行业务。读取场景提供的路由 spec，识别用户要进入哪个场景，提取可从对话直取的候选槽位，构造 Agent task 并追踪状态。

识别 场景召回、LLM 确认、歧义澄清。
派发 target_agent、handoff fields、agent_task_id。
追踪 路由 state、transcript、Agent event summary。
Agent Task scene_id + raw_message + routing_slots
↓
Agent Event waiting / running / completed / failed
场景执行层 Execution Agents

按场景自己的执行 Skill 完成业务流程。这里才处理业务提槽、风控、确认、幂等、业务 API 调用和异常处理。

Transfer Agent 转账提槽、确认、风控、转账 API。
Fund Agent 基金查询、申购试算、风险揭示。
Balance Agent 余额查询、账户信息、结果解释。
这张分层图是最重要的边界图：Router 处在助手和场景执行 Agent 中间，负责“把用户请求正确送到哪个场景”，不负责“替场景完成业务流程”。

代码负责什么

保存路由状态、组装上下文、读取场景提供的 spec、识别意图、抽取可直取槽位、派发执行 Agent、记录路由审计、压缩裁剪。

Spec 负责什么

Router 侧 spec 由场景提供，描述触发条件、目标 Agent、可由 Router 直取的槽位、派发字段和 reference；业务步骤、确认、风控、业务 API 由场景执行 Agent 的 spec 定义。

二、Router 和场景执行 Agent 的边界

Router 应该负责

识别意图、做必要澄清、读取场景提供的路由 spec、提取可从对话直取的候选槽位、选择目标 Agent、派发任务、记录路由 transcript、追踪 Agent 状态。

识别 派发 追踪 路由审计

场景执行 Agent 应该负责

业务提槽补全、业务校验、风控、确认、幂等、业务 API 调用、异常处理和最终业务结果。执行 Skill 由场景提供，用来说明对应执行 Agent 怎么工作。

业务流程 确认 风控 业务 API
Router 可以提槽，但槽位 schema 必须由场景 spec 提供；Router 提取出来的是派发 hints，执行 Agent 仍然拥有最终业务校验权。

三、四类事实不要混在一起

结构化状态

当前候选场景、目标 Agent、路由级 slots、派发状态、Agent task_id、最近回调状态。

必须持久化 路由事实源

Transcript

用户原文、路由决策、提取槽位、派发请求、Agent 回调、compaction event。

可审计 可回放

Spec / Reference

agent.md、场景路由 spec、槽位 spec、Agent 派发契约和说明文档。它们是可重载内容，不是每轮都塞全文。

按需加载 带版本/hash

Prompt View

本轮真正给模型看的短视图，由 ContextBuilder 按预算组装。

可裁剪 可解释

四、渐进加载不是一句话，是分层接口

L0 Agent Rules 短 agent.md，每轮可进 prompt
›
L1 场景索引 只读场景 metadata，不读全文
›
L2 场景路由 Spec 选中场景后读取派发契约和 slot spec
›
L3 场景说明 / Reference 识别或提槽需要时才读片段

错误方式

每轮 prompt =
agent.md 全文
+ 所有 skills/*.md 全文
+ 所有 references/*.md 全文
+ 全部历史消息
+ 全部 Agent 回调和业务结果

正确方式

每轮 prompt =
短 agent rules
+ 当前路由 session state
+ 最近几轮原文
+ 候选场景 metadata
+ 场景提供的可直取 slot spec
+ 目标 Agent 派发契约

五、完整例子：用户说“给张三转5000块”

掌银助手发起请求

上游传入 `session_id`、用户原文、用户画像、页面上下文和可派发的 Agent 信息。Router 不直接调用转账、申购等业务 API。

加载或创建 Session State

如果是新会话，`active_scene=null`、`routing_slots={}`。如果已有未完成路由或已派发任务，先恢复目标 Agent、task_id 和路由状态。

写入 Transcript

保存 `turn_id`、用户原文、请求快照引用、后续路由决策和派发事件。Transcript 是审计和回放来源，但不会全量进入 prompt。

加载短 Agent Rules

读取 `agent.md` 中稳定规则：Router 只做识别、澄清、提取可直取槽位、派发和追踪；业务确认和业务执行必须交给场景 Agent。

加载场景索引

只加载场景 metadata，例如 `transfer` 的名称、描述、目标 Agent、风险级别、关键词、hash。不读取全部场景说明正文。

识别或恢复场景

新会话用场景索引匹配到 `transfer`。如果 session 里已有 `active_scene=transfer` 或已派发 `agent_task_id`，则直接恢复，不从头识别。

读取场景路由 Spec

选中 `transfer` 后才读场景提供的路由 spec，拿到目标 `transfer-agent`、可由 Router 直取的 slot schema、派发字段和 reference 列表。

按当前 step 读取 Reference

如果识别或提槽需要业务说明，读取场景 reference 的必要片段。例如“收款人”和“金额”如何从自然语言中直取。完整业务执行规则仍由场景 Agent 持有。

提取 Router 可直取 slots

根据场景提供的 slot spec，从“给张三转5000块”提取候选 `recipient=张三`、`amount=5000`。这些是派发 hints，不替代执行 Agent 的业务校验。

构造 Agent Task

Router 把原始用户输入、路由槽位 hints、用户画像快照、页面上下文和场景 ID 打包成任务，不调用 `risk_check` 或 `transfer`。

派发给执行 Agent

通过 AgentDispatchGateway 调用 `transfer-agent`。`transfer-agent` 后续负责业务提槽补全、风控、确认、转账 API 和异常处理。

保存派发状态

Runtime 保存 `agent_task_id`、目标 Agent、派发摘要和当前状态，例如 `dispatched` 或 `waiting_agent`，用于多轮续接和回调关联。

后续用户消息的流转

如果任务已交给 `transfer-agent`，后续“确认”“改成3000”“取消”等消息应路由到该 Agent 或由 Agent 回调驱动，而不是 Router 自己解释业务确认。

接收 Agent 进度或完成

Router 记录 Agent 状态回调，例如 waiting_user_input、running、completed、failed。业务结果详情由 Agent 返回，Router 只负责传递和追踪。

完成并清理路由工作集

Agent 完成后，Router 清理 active routing state，保留派发摘要和结果摘要。场景 spec、reference 原文、Agent 详细过程不再进入后续 prompt。

八、和 OpenClaw / Hermes 的关系

借鉴它们

Skill 只默认放索引，全文按需读；历史做 transcript；旧上下文做 compaction/pruning；memory 和 reference 通过工具按需检索。

不能照搬它们

金融业务不能让 Router 或模型直接执行副作用。确认、幂等、风控、业务 API 应由场景执行 Agent 处理；Router 只校验路由和派发边界。
