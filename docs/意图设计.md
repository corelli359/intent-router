# 意图设计

## 1. 文档目标
本文档用于收敛意图路由在“任务未完成时的续轮处理”设计，重点回答以下问题：

- `ishandover = false` 时，Router 应如何处理下一轮用户输入
- 如何区分“继续当前意图”“切换意图”“同意图内切换目标”
- 多轮对话中，上下文应如何分层，避免旧任务污染新任务
- 对于转账等敏感动作，前端、Router、Agent 如何通过确认卡片闭环

本文档是对现有 PRD 的补充，聚焦运行时设计，不替代意图注册、部署和通用 SSE 协议文档。

## 2. 设计原则

### 2.1 Router 只管理任务生命周期，不管理业务语义
Router 的职责是：

- 判断当前未完成任务是否继续
- 判断用户是否取消当前任务
- 判断用户是否明确切换到新意图
- 维护任务状态、SSE 事件、上下文边界和任务关闭标记

Router 不应负责判定“转账目标是否变化”“预约对象是否变化”这类领域语义。此类语义应由对应 intent agent 自己理解。

### 2.2 只要任务未交接完成，默认先解释为续轮补充
当某个任务的最新有效状态仍表示“未完成交接”时，下一轮用户输入默认应被解释为对该任务的补充，而不是直接重新进入全局意图识别。

实现层应以任务状态为权威，`ishandover` 作为协议语义辅助：

- `status = waiting_user_input` 或 `status = waiting_confirmation`：当前任务仍为 open task
- `ishandover = false`：表示当前任务尚未闭环，通常对应 open task
- `status = completed / failed / cancelled` 且 `ishandover = true`：任务已关闭

注意：

- 不应仅凭任意中间流式消息中的 `ishandover = false` 判断任务仍在等待用户输入
- 应以 Router 持久化后的任务状态作为最终依据

### 2.3 默认继续，明确证据才切换
存在 open task 时，Router 对下一轮输入的默认策略应为：

- 有合理可能是在补当前槽位：继续当前任务
- 用户明确取消：关闭当前任务
- 用户明确提出新诉求：切换意图

不能因为用户输入中包含“卡号”“手机号后四位”“金额”等通用要素，就直接重新做全局意图识别并切走当前任务。

## 3. 任务状态与关闭标记

### 3.1 open task
当前设计建议将以下状态视为 open task：

- `created`
- `queued`
- `dispatching`
- `running`
- `resuming`
- `waiting_user_input`
- `waiting_confirmation`

其中真正允许用户续轮输入的主要是：

- `waiting_user_input`
- `waiting_confirmation`

### 3.2 closed task
以下状态视为 closed task：

- `completed`
- `failed`
- `cancelled`

### 3.3 关闭原因
建议为 closed task 增加 `closure_reason`，用于界定上下文边界与后续分析：

- `completed`
- `failed`
- `cancelled_by_user`
- `switched_intent`
- `replaced_by_new_target`
- `expired`

这样在“先查余额，再转账”这类场景中，前一个任务结束后即可明确标记为 closed，不再参与后续转账任务的续轮补槽。

## 4. 上下文分层

### 4.1 设计目标
上下文不能是一个不断累加、无边界污染的大桶。建议拆为以下四层：

### 4.2 Session Transcript
完整会话原始消息流，仅用于：

- 审计
- 调试
- 极端情况下的全局回溯

它不应直接被所有 agent 无差别拿去补槽。

### 4.3 Active Task Working Memory
当前 open task 的工作记忆，是续轮处理的第一优先级上下文，应包含：

- 当前意图编码
- 当前已收集槽位
- 当前缺失槽位
- 上次追问内容
- 上次确认卡片内容
- 当前 interaction mode
- 当前 task version

只要任务未关闭，用户的新输入默认围绕这一层解释。

### 4.4 Closed Task Summary
每个已关闭任务写入一份摘要，包含：

- `intent_code`
- 结果摘要
- 关键槽位摘要
- `closure_reason`
- 关闭时间

该层默认不参与当前任务的补槽，只用于：

- 展示历史结果
- 在明确需要时做全局对话理解

例如：

- 用户先完成余额查询，再开始转账
- 转账补槽时，不应再把余额查询中的卡号和手机号后四位自动灌入转账任务

### 4.5 Customer Long-Term Memory
这是跨会话长期记忆，只能存放明确允许长期保留的信息，且需要按业务合规控制。

在 V1 中，不建议用它去补全敏感交易字段。

## 5. Waiting 态下的续轮决策

### 5.1 不再复用普通全局意图识别
当存在 open task 时，下一轮输入不应直接复用普通 intent recognizer。应引入专门的 waiting decision 流程，只输出三类决策：

- `continue_current`
- `switch_intent`
- `cancel_current`

如果需要更细粒度，可扩展：

- `continue_with_target_change`

但该结果更适合由 agent 返回，而不是由 Router 直接做业务判断。

### 5.2 Waiting Decision 输入
建议 waiting decision 至少读取：

- 当前 open task 的 `intent_code`
- 当前 open task 的已收集槽位
- 当前缺失槽位
- 当前交互模式
- 最近 3 到 5 轮对话
- 本轮用户输入

### 5.3 Waiting Decision 输出原则

#### 继续当前任务
满足以下任一条件时，优先判定为 `continue_current`：

- 本轮输入看起来是在补当前缺失字段
- 本轮输入与当前意图强相关，且没有明确新诉求
- 无法确定是否切换时，保守继续当前任务

#### 取消当前任务
满足以下条件时，可判定为 `cancel_current`：

- 用户明确表达“取消”“算了”“不用了”“停止当前这个”
- 用户确认放弃当前任务

#### 切换意图
仅当以下条件成立时，才允许 `switch_intent`：

- 用户明确提出新的意图诉求
- 新诉求不是在补当前任务缺失字段
- 当前任务应被关闭，并写入 `closure_reason = switched_intent`

示例：

- 当前在转账补槽，用户输入“不要转账了，帮我查余额”
- 这是明确切意图，应关闭转账任务，再进入全局意图识别

### 5.4 最关键的约束
在 waiting 态下，不能因为“卡号 + 手机号后四位”也可能匹配余额查询，就自动把转账任务切到余额查询。只要它们仍可能是当前转账任务的补槽，就应继续当前任务。

## 6. 同意图内的目标变化

### 6.1 目标变化不由 Router 硬判
“同意图但目标变化”属于业务语义，Router 不应泛化处理。应由对应 agent 自行判定关键目标字段是否发生冲突。

例如转账 intent 的关键目标字段可以定义为：

- `recipient_name`
- `recipient_card_number`
- `recipient_phone_last_four`

若当前任务中这些字段已明确，而用户新输入又提供了冲突值，则应视为“目标变化候选”。

### 6.2 目标变化后的建议策略
当 agent 检测到同意图内目标变化时，不建议直接覆盖老值。建议返回：

- `status = waiting_confirmation`
- `ishandover = false`
- 目标变更确认卡片

由用户明确选择：

- 确认替换为新目标
- 保留原目标
- 取消本次交易

### 6.3 槽位覆盖原则
建议统一为“精确覆盖”而不是“模糊清空”：

- 本轮明确提供了哪个槽位，只覆盖哪个槽位
- 没提到的已确认槽位保留
- 只有当用户确认目标变化后，才批量替换目标相关槽位

## 7. 确认卡片设计

### 7.1 设计目标
对于转账等敏感动作，仅靠自然语言补槽不够稳。应支持 agent 返回结构化确认卡片，由前端渲染，用户通过明确操作完成确认。

### 7.2 交互模式
建议任务增加 `interaction_mode`：

- `collect_slots`
- `confirm_card`

当任务进入确认卡片阶段时：

- `status = waiting_confirmation`
- `ishandover = false`
- `interaction_mode = confirm_card`

### 7.3 SSE 载荷约定
建议在任务事件中增加 `payload.interaction`，例如：

```json
{
  "event": "task.waiting_confirmation",
  "task_id": "task_xxx",
  "session_id": "session_xxx",
  "intent_code": "transfer_money",
  "status": "waiting_confirmation",
  "message": "请确认转账信息",
  "ishandover": false,
  "payload": {
    "interaction": {
      "type": "confirm_card",
      "card_code": "transfer_review_v1",
      "version": 1,
      "title": "请确认转账信息",
      "fields": [
        {"key": "recipient_name", "label": "收款人", "value": "张三"},
        {"key": "recipient_card_number", "label": "收款卡号", "value": "622202****9999"},
        {"key": "recipient_phone_last_four", "label": "手机号后4位", "value": "1234"},
        {"key": "amount", "label": "转账金额", "value": "5000"}
      ],
      "actions": [
        {"code": "confirm", "label": "确认"},
        {"code": "edit", "label": "修改"},
        {"code": "cancel", "label": "取消"}
      ],
      "confirm_token": "opaque_token"
    }
  }
}
```

约束：

- 卡号等敏感字段必须脱敏展示
- `confirm_token` 必须和当前任务版本绑定，防止旧卡片误操作

### 7.4 Frontend 责任
前端负责：

- 根据 `payload.interaction` 渲染卡片
- 区分自由输入态和确认卡片态
- 在用户点击按钮后，发起结构化 action 请求，而不是伪造成普通文本消息

### 7.5 Router 责任
Router 负责：

- 透传 agent 返回的卡片信息
- 持久化当前任务的确认状态、任务版本和 `confirm_token`
- 校验 action 是否仍针对当前有效任务版本

Router 不负责解释卡片业务语义。

### 7.6 Agent 责任
Agent 负责：

- 生成卡片内容
- 定义当前卡片允许的 action
- 在收到 action 后决定继续执行、回到补槽，或取消任务

## 8. 结构化 Action 接口建议

### 8.1 不建议按钮点击继续走纯文本消息
确认卡片的按钮操作属于结构化交互，建议单独定义接口：

`POST /api/router/sessions/{session_id}/actions`

示例：

```json
{
  "task_id": "task_xxx",
  "action_code": "confirm_transfer",
  "confirm_token": "opaque_token",
  "payload": {
    "decision": "confirm"
  }
}
```

### 8.2 Action 请求的处理原则
Router 收到 action 后：

- 校验 `task_id`
- 校验 `confirm_token`
- 校验当前任务是否仍为 open task
- 校验卡片版本是否仍为最新

校验通过后再交给 agent 继续处理。

## 9. 一个推荐的运行时闭环

### 9.1 先查余额，再转账

1. 用户发起余额查询
2. 余额 agent 收集完参数并完成任务
3. Router 将余额任务标记为 `closed`
4. 用户发起转账
5. Router 创建新的转账 open task
6. 后续续轮只围绕转账 working memory 处理，不再混入余额任务槽位

### 9.2 转账过程中补槽

1. 用户说“帮我转账”
2. Agent 返回 `waiting_user_input`
3. 用户说“张三”
4. Router 因当前存在 open task，优先按续轮补槽解释
5. Agent 仅补 `recipient_name`
6. 用户再补“卡号……手机号后四位……”
7. Router 继续当前任务，不做全局切意图
8. 最后补金额后完成任务

### 9.3 转账过程中切意图

1. 当前处于转账 `waiting_user_input`
2. 用户说“不要转账了，帮我查余额”
3. Waiting decision 判定为 `switch_intent`
4. Router 关闭当前转账任务，标记 `closure_reason = switched_intent`
5. 再进入全局意图识别，创建余额任务

### 9.4 转账过程中切目标

1. 当前已收集 `recipient_name = 张三`
2. 用户后续说“改成给李四”
3. Router 仍继续当前转账任务
4. 转账 agent 识别出关键目标字段发生变化
5. Agent 返回 `waiting_confirmation` 和目标变更确认卡片
6. 用户确认后再替换目标槽位并继续执行

## 10. 推荐落地顺序

### 10.1 第一阶段
先解决当前线上最影响体验的问题：

- waiting 态不再直接复用普通全局意图识别
- 引入 `continue_current / switch_intent / cancel_current` 决策
- 默认继续当前任务，严格限制切意图

### 10.2 第二阶段
收敛上下文边界：

- 引入 active task working memory
- 引入 closed task summary
- 引入 `closure_reason`

### 10.3 第三阶段
为敏感意图引入确认卡片：

- 增加 `waiting_confirmation`
- 定义 `payload.interaction`
- 定义 `/actions` 接口
- 先在 `transfer_money` 上落地

## 11. 结论
本文档建议将当前问题收敛为一个简单原则：

- Router 只管理“当前 open task 是否继续”
- Agent 管理“当前 intent 内部的目标和确认”
- 已关闭任务必须做边界标记，避免污染当前任务
- 敏感动作通过结构化确认卡片闭环，而不是继续依赖纯自然语言猜测

这套设计的目标不是一次性覆盖所有复杂业务，而是先把多轮续轮、切意图、切目标和敏感确认这四个关键问题稳定下来。
