## 1. 背景与目标

当前项目的业务理解链路，已经清楚地分成三层：

1. 大类识别  
   例如缴费、转账、营销、理财。

2. 小类或叶子意图识别  
   例如缴费域下的电费、水费、燃气费；转账域本身可能就是叶子意图。

3. 叶子意图槽位填充  
   例如燃气缴费的户号、金额；转账的收款人、金额、卡号等。

此外，业务上还存在条件语义：

- 顺序条件：先查余额，再转账
- 阈值条件：如果余额大于 5000，就转 2000
- 并行条件：同时做两件事
- 上下文条件：如果汇率高于 7.2 再换汇

当前 graph runtime 已经具备执行编排能力，但还没有把“层级路由”和“条件治理”作为一等公民纳入运行时控制。

本方案的目标不是推翻现有 graph，而是把现有 V2 graph runtime 升级为一条更稳的理解与执行链路：

```text
用户消息
-> 大类路由
-> 小类/叶子意图路由
-> 叶子意图槽位抽取
-> 条件抽取
-> 槽位/条件联合校验
-> 受控 memory 预填充
-> 生成叶子执行 graph
-> Router 确认 graph 可执行
-> 下发 agent
```

补充说明：

- 条件必须单独设计，不能并入普通槽位逻辑
- 但条件不作为首批开发阻塞项
- 首批优先把“大类 -> 叶子 -> 槽位 -> Router 校验”的主链路做稳
- 条件作为独立增强工作流，在主链路稳定后接入

## 1.1 当前新增优化原则

在最近一轮运行时优化之后，Router 对“是否调用重型规划”增加了单独策略控制，不再把“建 graph”和“调用 LLM planner”绑定成一件事。

当前推荐原则是：

1. 单大类 / 单叶子意图，且消息里没有明显条件、顺序、并行、重复动作信号  
   默认不走重型 LLM planner，而是直接走确定性单节点编译，然后进入槽位填充。

2. 多大类 / 多叶子意图  
   默认进入 graph 规划，因为需要拆动作、排顺序、挂依赖关系。

3. 虽然只有一个大类，但句子里存在复杂 graph 信号  
   例如“先…再…/如果…就…/还要…/同时…”，仍然允许升级到重规划。

对应的运行时配置为：

```text
ROUTER_V2_PLANNING_POLICY=always|never|multi_intent_only|auto
```

其中推荐默认值是：

```text
ROUTER_V2_PLANNING_POLICY=auto
```

这样做的目标不是取消 graph，而是把 graph 能力留给真正需要 graph 的场景，把“大类识别 + 槽位填充”这条主链路从高成本规划中解耦出来。

## 2. 约束与边界

本方案默认以下边界不变：

1. 不考虑模型微调。
2. 不引入传统机器学习分类器或序列标注模型。
3. Router 负责理解、补槽、条件治理、图编译。
4. Agent 只负责：
   - 检查 Router 下发的执行参数是否满足业务前置条件
   - 执行业务
   - 返回执行结果或业务级补充信息
5. Graph 继续保留，但 Graph 的职责是“叶子执行编排”，不是“层级分类过程可视化”。

## 3. 对当前代码的判断

## 3.1 当前已有能力

当前代码已经具备以下可复用基础：

1. 意图已支持 `field_catalog`、`slot_schema`、`graph_build_hints`。
   - `backend/services/router-service/src/router_service/models/intent.py`
   - `backend/services/admin-service/src/admin_service/models/intent.py`

2. Graph 已支持节点、边、条件边、顺序/并行/条件执行。
   - `backend/services/router-service/src/router_service/core/v2_domain.py`
   - `backend/services/router-service/src/router_service/core/v2_graph_runtime.py`

3. 已存在槽位绑定、历史复用、推荐预填、确认态抬升。
   - `backend/services/router-service/src/router_service/core/slot_resolution_service.py`
   - `backend/services/router-service/src/router_service/core/slot_grounding.py`

4. 已存在 unified builder 和 legacy planner，可作为过渡阶段兼容能力。
   - `backend/services/router-service/src/router_service/core/v2_graph_builder.py`
   - `backend/services/router-service/src/router_service/core/v2_planner.py`

## 3.2 当前缺口

当前缺口主要有五类：

1. 意图注册模型仍是扁平 intent 模型，没有正式的层级字段。
   - 缺少 `domain_code`
   - 缺少叶子/非叶子边界
   - 缺少 domain 级示例与约束

2. “识别 + 建图 + 槽位”被压在单次 LLM 输出中，控制过度依赖 prompt。
   - 当前 unified builder 同时输出 `primary_intents`、`nodes`、`slot_memory`
   - 这使得 domain 路由、小类路由、槽位抽取之间缺少天然分段

3. Router 侧缺少独立的 slot validator 和 condition validator。
   - 现在更多是 LLM 生成后做轻量 normalize
   - Router 下发 agent 前没有真正的“可执行门”

4. 条件存在表达能力，但尚未形成条件专项治理链路。
   - 当前条件主要依赖 prompt 约束和 graph runtime 修补
   - 尚未独立出 condition extraction / validation / source resolution

5. memory 还是字符串化 recall，缺少结构化候选检索与冲突控制。
   - `LongTermMemoryStore` 当前返回 `list[str]`
   - 历史槽位按 `slot_key` 合并，易出现跨意图串槽

## 4. 目标架构

## 4.1 总体原则

目标架构遵循以下原则：

1. 先收窄，再抽取  
   先做 domain，再做 leaf intent，再抽具体槽位与条件。

2. 条件独立于执行槽位  
   执行参数放在 `node.slot_memory`，条件表达式放在 `edge.condition`，绝不混用。

3. memory 只做候选补全  
   memory 不直接当当前轮真值，只做受控候选。

4. Router 在下发 agent 前必须完成可执行校验  
   Agent 不应成为第一层发现“缺槽位”的地方。

5. Graph 只表达叶子执行关系  
   Graph 不增加“缴费分类节点”“理财分类节点”这种无执行意义的节点。

## 4.2 目标链路

目标链路拆成七个模块：

1. `DomainRouter`
2. `LeafIntentRouter`
3. `SlotExtractor`
4. `ConditionExtractor`
5. `UnderstandingValidator`
6. `GraphCompiler`
7. `ExecutionOrchestrator`

职责边界如下：

### DomainRouter

输入：

- 用户消息
- active domains
- 最近对话摘要

输出：

- `primary_domains`
- `candidate_domains`
- `source_fragments`

职责：

- 只判断消息属于哪些大类
- 不直接抽取叶子槽位
- 不输出 graph

### LeafIntentRouter

输入：

- 用户消息
- 已确定的 `primary_domains`
- 各 domain 下的 leaf intents

输出：

- `primary_leaf_intents`
- `candidate_leaf_intents`
- 每个 leaf intent 对应的 `source_fragment`

职责：

- 只在单个 domain 的子集内做 leaf 路由
- 如果某个 domain 下只有一个 leaf intent，可直接确定

### SlotExtractor

输入：

- 单个 leaf intent
- 该 leaf intent 的 `slot_schema`
- 与该 leaf intent 对应的 `source_fragment`
- 极少量结构化 memory candidates

输出：

- `slot_memory`
- `slot_bindings`
- `missing_slots`
- `ambiguous_slots`

职责：

- 只抽取执行槽位
- 不处理条件表达式

### ConditionExtractor

输入：

- 用户消息
- 已确定的 leaf intent 列表
- 当前 graph 语义上下文
- 各 intent 的上下文产出声明

输出：

- 条件关系清单
- 条件依赖源
- 条件证据片段
- 缺失条件源节点建议

职责：

- 单独抽取顺序、并行、条件依赖
- 不把阈值错误写入 `slot_memory`

### UnderstandingValidator

输入：

- leaf intents
- `slot_memory`
- `slot_bindings`
- condition draft
- memory candidates
- `slot_schema`
- condition source schema

输出：

- `validated_slots`
- `validated_conditions`
- `missing_required_slots`
- `ambiguous_fields`
- `conflict_items`
- `needs_confirmation`
- `can_dispatch`

职责：

- Router 侧的可执行门
- 决定是否进入追问、确认或下发 agent

### GraphCompiler

输入：

- 已验证的 leaf intents
- 已验证的 slots
- 已验证的 conditions

输出：

- 叶子执行 graph

职责：

- 自动补条件依赖源节点
- 组织顺序、并行、条件边
- 不再承担开放式路由任务

### ExecutionOrchestrator

输入：

- 已验证 graph

输出：

- 执行态 session、node、task 演进

职责：

- 维持现有 graph runtime 能力
- 不再承担第一层理解工作

## 5. 数据模型改造

## 5.1 意图注册模型

需要在 Admin 和 Router 共用的 intent 模型中新增层级字段。

建议新增字段：

```python
class IntentPayload(BaseModel):
    domain_code: str = Field(default="", max_length=128)
    domain_name: str = Field(default="", max_length=256)
    domain_description: str = Field(default="", max_length=2000)
    is_leaf_intent: bool = True
    parent_intent_code: str = Field(default="", max_length=128)
    routing_examples: list[str] = Field(default_factory=list)
```

字段解释：

- `domain_code`  
  大类编码，必须稳定。例如 `payment`、`transfer`、`marketing`、`wealth_management`。

- `domain_name`  
  大类中文名，例如“缴费”“转账”。

- `domain_description`  
  domain 路由阶段给模型的说明，不等同于 leaf intent 的 description。

- `is_leaf_intent`  
  当前 intent 是否可执行。第一阶段建议所有注册 intent 仍然是叶子 intent，不引入非叶子 intent 实体。

- `parent_intent_code`  
  预留给未来三级分类，不强制启用。

- `routing_examples`  
  用于 domain/leaf 路由阶段，而不是 agent 执行阶段。

实施文件：

- `backend/services/admin-service/src/admin_service/models/intent.py`
- `backend/services/router-service/src/router_service/models/intent.py`
- `backend/services/admin-service/src/admin_service/api/schemas.py`
- `backend/services/admin-service/src/admin_service/storage/*`
- `backend/services/router-service/src/router_service/catalog/*`

## 5.2 条件源与上下文产出模型

当前已有 `graph_build_hints.provides_context_keys`，建议升级成更正式的上下文产出声明。

建议新增：

```python
class IntentConditionOutputDefinition(BaseModel):
    context_key: str
    label: str = ""
    description: str = ""
    value_type: SlotValueType = SlotValueType.STRING
    aliases: list[str] = Field(default_factory=list)

class IntentGraphBuildHints(BaseModel):
    ...
    condition_outputs: list[IntentConditionOutputDefinition] = Field(default_factory=list)
```

说明：

- `condition_outputs` 用于声明某个 intent 能产出什么条件字段
- 例如余额查询能产出 `balance`
- 账单查询能产出 `due_amount`
- 汇率查询能产出 `exchange_rate`

`provides_context_keys` 可以保留兼容，但后续主要消费 `condition_outputs`。

## 5.3 槽位 schema 约束升级

现有 `IntentSlotDefinition` 已较完整，但要明确使用规则：

1. 同类型多槽位必须依靠 `role` 区分。
2. 执行槽位与条件槽位不能混用同一 `slot_key`。
3. 高风险槽位必须明确 `confirmation_policy`。
4. `bind_scope=CONDITION_OPERAND` 只用于条件操作数，不进入执行 `slot_memory`。

第一阶段不建议让多数 intent 混用 `NODE_INPUT` 和 `CONDITION_OPERAND`，更建议：

- 叶子 intent 的 `slot_schema` 只定义执行参数
- 条件操作数字段由 `condition_outputs` 提供

## 5.4 Memory 结构化模型

当前 `LongTermMemoryStore` 需要从字符串列表升级为结构化记录。

建议模型：

```python
class MemorySlotRecord(BaseModel):
    cust_id: str
    slot_key: str
    value: Any
    value_type: str
    domain_code: str = ""
    intent_code: str = ""
    confirmed: bool = False
    sensitivity: str = "normal"
    source_session_id: str = ""
    source_task_id: str = ""
    updated_at: datetime
    ttl_seconds: int = 0
    confidence: float = 0.0
```

必须支持的过滤条件：

- 按 `slot_key`
- 按 `intent_code`
- 按 `domain_code`
- 按 `confirmed`
- 按 TTL 是否过期
- 按敏感级别

`GraphNodeState` 建议补充以下路由与校验元数据：

class GraphNodeState(BaseModel):

## 6. Router 模块改造

## 6.1 新增 DomainRouter

建议新增文件：

- `backend/services/router-service/src/router_service/core/domain_router.py`

建议接口：

```python
class DomainRouteMatch(BaseModel):
    domain_code: str
    confidence: float
    reason: str
    source_fragment: str | None = None

class DomainRoutingResult(BaseModel):
    primary: list[DomainRouteMatch]
    candidates: list[DomainRouteMatch]
```

路由规则：

1. 只在 domain 层判断，不输出 intent_code。
2. 如果用户一句话中存在多个独立目标，可返回多个 `primary domain`。
3. 如果只是同一业务动作的多个参数，不得拆成多个 domain。
4. domain router 不得消费大段 long-term memory。

结构化输出要求：

- 必须走 strict schema
- 不再只依赖流式文本 JSON 解析

## 6.2 新增 LeafIntentRouter

建议新增文件：

- `backend/services/router-service/src/router_service/core/leaf_router.py`

建议接口：

```python
class LeafIntentRouteItem(BaseModel):
    domain_code: str
    intent_code: str
    confidence: float
    reason: str
    source_fragment: str | None = None

class LeafIntentRoutingResult(BaseModel):
    primary: list[LeafIntentRouteItem]
    candidates: list[LeafIntentRouteItem]
```

路由规则：

1. 每个 domain 独立判 leaf intent。
2. 若某个 domain 只有一个 leaf intent，则直接命中，不调用模型。
3. `source_fragment` 尽量与当前 leaf intent 对应，便于后续 slot 和 condition 提取。

## 6.3 新增 SlotExtractor

建议新增文件：

- `backend/services/router-service/src/router_service/core/slot_extractor.py`

建议接口：

```python
class SlotExtractionResult(BaseModel):
    slot_memory: dict[str, Any] = Field(default_factory=dict)
    slot_bindings: list[SlotBindingState] = Field(default_factory=list)
    missing_slots: list[str] = Field(default_factory=list)
    ambiguous_slots: list[str] = Field(default_factory=list)
```

工作方式：

1. 一次只针对一个 leaf intent。
2. 输入只包含该 leaf intent 的 `slot_schema`。
3. 允许少量结构化 memory candidates 作为 side input。
4. 输出不包含条件表达式。

关键约束：

- 没有证据的高风险槽位，不直接进入最终 `slot_memory`
- 不允许把 history 中的值直接冒充当前轮槽位

## 6.4 新增 ConditionExtractor（后置）

建议新增文件：

- `backend/services/router-service/src/router_service/core/condition_extractor.py`

建议接口：

```python
class ConditionDraft(BaseModel):
    source_intent_code: str | None = None
    target_intent_code: str
    relation_type: Literal["sequential", "parallel", "conditional"]
    left_key: str | None = None
    operator: str | None = None
    right_value: Any | None = None
    source_text: str | None = None

class ConditionExtractionResult(BaseModel):
    items: list[ConditionDraft] = Field(default_factory=list)
    missing_source_nodes: list[str] = Field(default_factory=list)
```

必须满足的规则：

1. 顺序、并行、条件关系单独抽取。
2. 阈值类数值不进入 `node.slot_memory`。
3. 当条件依赖字段不存在对应 source node 时，标记为 `missing_source_nodes`，交给 compiler 自动补图。
4. 条件也必须尽量产出 `source_text`。

## 6.5 新增 UnderstandingValidator

建议新增文件：

- `backend/services/router-service/src/router_service/core/understanding_validator.py`

内部拆分两个子组件：

- `slot_validator.py`
- `condition_validator.py`（后置）

建议输出：

```python
class UnderstandingValidationResult(BaseModel):
    validated_slot_memory: dict[str, Any] = Field(default_factory=dict)
    validated_slot_bindings: list[SlotBindingState] = Field(default_factory=list)
    validated_conditions: list[ConditionDraft] = Field(default_factory=list)
    missing_required_slots: list[str] = Field(default_factory=list)
    ambiguous_items: list[str] = Field(default_factory=list)
    conflict_items: list[str] = Field(default_factory=list)
    needs_confirmation: bool = False
    can_dispatch: bool = False
```

核心职责：

1. 判定 leaf intent 是否已具备可执行参数。
2. 判定是否需要 Router 追问。
3. 判定是否需要用户确认。
4. 判定 graph 是否可以编译并下发。

实施说明：

- 首期先落地 `slot_validator.py`
- `condition_validator.py` 保留为后续扩展项

## 7. 条件专项设计（独立工作流，后置实施）

条件是本方案的一等治理对象，但不作为第一批开发阻塞项。

实施顺序建议：

1. 先完成层级路由、叶子槽位抽取、Router 可执行门。
2. 再引入条件抽取、条件校验和自动补图。

这样安排的原因是：

1. 当前最核心的问题仍然是 leaf intent 没有稳定收敛，以及 Router 侧缺少独立的补槽校验。
2. 条件能力建立在 leaf intent 已确定、执行槽位已稳定、graph 边界已清晰的前提上。
3. 如果过早把条件接进主链路，会把调试维度从“分类 + 槽位”直接升级为“分类 + 槽位 + 图关系 + 条件推理”，复杂度会显著抬高。

因此，条件的策略是：

- 设计现在就要定清楚
- 数据模型现在就要预留
- 实际编码放到主链路稳定之后

## 7.1 设计目标

条件专项要解决四类问题：

1. 条件识别  
   用户有没有表达顺序、并行、阈值、触发规则。

2. 条件绑定  
   条件约束哪个目标节点，依赖哪个 source node。

3. 条件校验  
   条件字段、比较符、阈值、source node 是否都合法。

4. 条件补图  
   用户没有明确说出条件源节点时，Router 能否自动补出 producer node。

## 7.2 条件类型划分

建议统一支持四类条件语义：

1. `sequential`
   - 示例：“先查余额，再转账”

2. `parallel`
   - 示例：“同时查余额和查账单”

3. `conditional`
   - 示例：“如果余额大于 5000，就转 2000”

4. `contextual-conditional`
   - 示例：“如果汇率高于 7.2，再换汇”

工程上不需要新增第四种 `relation_type`，它本质上仍属于 `conditional`，只是条件左值来自上下文型 producer node。

## 7.3 条件与执行槽位彻底分仓

必须执行以下强约束：

1. `node.slot_memory` 只存执行参数。
2. `edge.condition` 只存条件表达式。
3. 条件阈值不允许写入任何执行槽位。
4. 如果发现同一个数值同时被绑定为条件阈值和执行参数，必须进入 `ambiguous/conflict`。

典型例子：

输入：

```text
如果余额大于 5000，就给妈妈转 2000
```

正确结果：

- `query_account_balance` 提供 `balance`
- `transfer_money.amount = 2000`
- 条件为 `balance > 5000`

错误结果：

- `transfer_money.amount = 5000`
- 或把 `5000` 同时写入 `amount` 和条件阈值

## 7.4 条件数据模型思路

建议最小条件草稿模型如下：

```python
class ConditionDraft(BaseModel):
    source_intent_code: str | None = None
    target_intent_code: str
    relation_type: Literal["sequential", "parallel", "conditional"]
    left_key: str | None = None
    operator: Literal[">", ">=", "==", "<", "<="] | None = None
    right_value: Any | None = None
    source_text: str | None = None
```

字段语义：

- `source_intent_code`
  条件值来自哪个 source node。允许为空，交给 compiler 自动补图。

- `target_intent_code`
  条件作用在哪个目标节点。

- `relation_type`
  表示顺序、并行或条件关系。

- `left_key`
  条件左侧字段，例如 `balance`、`exchange_rate`、`due_amount`。

- `operator`
  条件比较符。

- `right_value`
  条件阈值。

- `source_text`
  条件证据片段。

## 7.5 条件源显式注册

必须明确哪些 intent 能提供条件源字段。

示例：

- `query_account_balance` 提供 `balance`
- `query_credit_card_repayment` 提供 `due_amount`、`minimum_due`
- `exchange_forex_quote` 提供 `exchange_rate`

如果没有显式注册，就不允许作为条件源。

## 7.6 条件提取的设计思路

ConditionExtractor 应独立于 SlotExtractor，原因如下：

1. 条件里的值与执行槽位值很容易冲突，尤其是金额、日期、姓名、卡号。
2. 条件抽取依赖图上下文，它是节点间关系问题，不是单个 intent 的局部问题。
3. 条件更适合后处理，先有 leaf intent 和执行槽位，再去判断它们之间的约束关系会更稳。

推荐顺序：

1. 先完成 domain route
2. 再完成 leaf route
3. 再完成 slot extraction
4. 最后独立做 condition extraction

不建议一开始就同时抽 leaf intent、slot 和 condition。

## 7.7 自动补条件源节点

GraphCompiler 必须支持补隐式节点。

示例：

输入：

```text
如果余额大于 5000，就给弟弟转 2000
```

用户只说了转账动作，但条件依赖余额。Router 需要自动补：

1. `query_account_balance`
2. `transfer_money`
3. `edge.condition = balance > 5000`

这类能力属于 Router graph compiler，不属于 agent。

## 7.8 条件校验思路

ConditionValidator 至少要做以下检查：

1. `target_intent_code` 是否存在于当前 graph seed 中。
2. `source_intent_code` 若存在，是否真的能产出 `left_key`。
3. `left_key` 是否在 `condition_outputs` 中声明。
4. `operator` 是否合法。
5. `right_value` 的类型是否与 `left_key` 匹配。
6. 条件值是否错误落入某个 node 的 `slot_memory`。
7. 条件是否缺少 source node，若缺少则标记待补图。
8. 多个条件是否互相冲突。

建议输出状态：

- `valid`
- `missing_source`
- `ambiguous`
- `conflict`
- `invalid`

## 7.9 条件禁止 history 自动注入

默认策略：

1. 条件阈值不允许从 history 自动注入。
2. 条件操作对象不允许从 history 自动注入。
3. 条件时间窗口不允许从 history 自动注入。
4. 若未来允许少数条件来自历史，也必须进入确认态。

理由：

- 条件值强时效
- 错用历史条件的代价高
- 条件本质上比普通槽位更依赖当前轮语义

## 7.10 条件 Router 追问

当出现以下情况时，必须由 Router 追问，而不是直接下发：

1. 条件字段缺失  
   例如“如果够的话就转”

2. 条件阈值缺失  
   例如“如果余额高一点就转”

3. 条件源歧义  
   例如“如果它余额够就转”，但前文存在多个账户

4. 条件与执行参数冲突  
   例如两个金额无法判断谁是阈值谁是执行金额

## 7.11 条件设计示例

### 示例 1：顺序关系

输入：

```text
先查余额，再给妈妈转 500
```

目标理解：

- leaf intents：
  - `query_account_balance`
  - `transfer_money`
- `transfer_money.amount = 500`
- edge：
  - `query_account_balance -> transfer_money`
  - `relation_type = sequential`

说明：

- 这不是阈值条件，只是顺序关系
- 不需要 `left_key/operator/right_value`

### 示例 2：显式阈值条件

输入：

```text
如果余额大于 5000，就给弟弟转 2000
```

目标理解：

- leaf intents：
  - `query_account_balance`
  - `transfer_money`
- `transfer_money.amount = 2000`
- 条件：
  - `left_key = balance`
  - `operator = >`
  - `right_value = 5000`

说明：

- `5000` 是条件阈值
- `2000` 是执行金额
- 若没有余额节点，要自动补 `query_account_balance`

### 示例 3：上下文型条件

输入：

```text
如果汇率高于 7.2，再换 300 美元
```

目标理解：

- leaf intents：
  - `query_exchange_rate` 或等价 quote intent
  - `exchange_forex`
- `exchange_forex.amount = 300`
- 条件：
  - `left_key = exchange_rate`
  - `operator = >`
  - `right_value = 7.2`

说明：

- `7.2` 不是换汇金额
- graph compiler 需要补能产出 `exchange_rate` 的 source node

### 示例 4：条件不完整，需要 Router 追问

输入：

```text
如果够的话就给妈妈转 500
```

目标理解：

- 已知 leaf intent：`transfer_money`
- 已知执行槽位：`amount = 500`
- 条件不完整，缺失：
  - `left_key`
  - `right_value`

Router 行为：

- 不创建 task
- 当前 node 进入 `WAITING_USER_INPUT`
- 追问：“你说的‘够’是指余额大于多少，还是别的条件？”

### 示例 5：条件与执行金额相同但语义不同

输入：

```text
如果余额超过 2000，就转 2000
```

目标理解：

- 阈值金额：`2000`
- 执行金额：`2000`

说明：

- 允许两个值数值相同，但语义位置不同
- 必须通过 `source_text` 和绑定位置区分
- 不能因为两个值一样，就只保留一个绑定

## 7.12 条件实施策略

条件不先做，但要现在就预留好三类能力：

1. 数据模型预留
   - `condition_outputs`
   - `ConditionDraft`
   - graph edge condition schema

2. prompt 和 validator 边界预留
   - 明确条件不属于普通 `slot_memory`
   - 明确 Router 后续会有 condition validator

3. graph compiler 接口预留
   - 编译器要接受“已验证条件草稿”作为输入

这样做的好处是：

- 第一批开发不必实现完整条件链路
- 但第二批接条件时，不需要推翻主链路设计

## 8. Memory 策略

## 8.1 基本原则

memory 的职责不是参与大类识别，而是参与叶子级补槽。

正确顺序是：

```text
先确定 leaf intent
-> 再检索该 intent 允许复用的 memory candidates
-> 再做 slot validation
```

不允许：

```text
先把大量历史记忆塞进大类路由
-> 再让模型顺带判断意图
```

## 8.2 允许复用的槽位

更适合 history 复用的类型：

- 默认卡号
- 手机号后四位
- 常用地址
- 稳定联系人称谓
- 默认币种

## 8.3 默认禁止自动复用的槽位

默认禁用 history 自动注入的类型：

- 当前交易金额
- 当前订单号
- 当前账单号
- 当前日期/时间
- 当前动作对象
- 所有条件阈值

## 8.4 检索顺序

建议检索顺序：

1. 同 leaf intent 已确认值
2. 同 domain 已确认值
3. 全局共享字段，但必须过严格 gating

## 9. GraphCompiler 改造

## 9.1 重新定义职责

改造后的 `GraphCompiler` 只承担以下职责：

1. 把已确定的 leaf intent 组装成 graph
2. 绑定已验证的 slot memory
3. 在后续阶段绑定已验证的条件关系
4. 在后续阶段自动补条件依赖源节点
5. 决定 graph 是否进入确认态

不再承担：

- 开放式 domain 路由
- 开放式 leaf intent 识别
- 开放式槽位抽取

## 9.2 与当前 unified builder 的关系

处理建议：

1. 保留现有 unified builder 作为兼容链路。
2. 新增一种理解模式，例如：

```text
ROUTER_V2_UNDERSTANDING_MODE=flat|hierarchical
```

3. 当 `ROUTER_V2_PLANNING_POLICY=always` 时，允许继续启用 unified builder。
4. 当 `ROUTER_V2_PLANNING_POLICY=auto|multi_intent_only|never` 时，首轮链路优先回到“识别 -> 策略化编译”，避免在简单单意图场景里仍然先走一遍重建图。

## 9.3 Graph 生成顺序

推荐编译顺序：

1. 基于 validated leaf intents 建初始节点
2. 对每个节点挂载 validated `slot_memory`
3. 无条件阶段先只处理顺序/并行关系
4. 条件阶段再补充 conditional edge 与 producer node 自动补图
5. 重新排序 node position
6. 根据确认策略、history 使用、条件复杂度决定 graph status

## 10. Orchestrator 改造

## 10.1 Router 侧 pre-dispatch gate

当前 `_create_task_for_node()` 前缺少 Router 的强 gate。必须补：

1. 没有通过 `UnderstandingValidator` 的节点，不创建 task。
2. `missing_required_slots` 时，node 进入 `WAITING_USER_INPUT`。
3. `needs_confirmation` 时，node 或 graph 进入确认态。
4. 只有 `can_dispatch=true` 才允许创建 task。

实施文件：

- `backend/services/router-service/src/router_service/core/v2_orchestrator.py`

## 10.2 Router 追问闭环

Router 需要能够在未创建 task 的情况下维持当前 node 的追问状态。

要求：

1. 记录当前等待补充的是哪个 leaf intent
2. 用户补充后首期只重跑该 node 的 slot extraction，后续扩展到 condition extraction
3. 不重跑整张 graph

## 13. 风险与兼容策略

## 13.1 兼容策略

短期内建议保留以下双轨：

1. `ROUTER_V2_GRAPH_BUILD_MODE=legacy|unified`
2. `ROUTER_V2_UNDERSTANDING_MODE=flat|hierarchical`
3. `ROUTER_V2_PLANNING_POLICY=always|never|multi_intent_only|auto`

推荐策略：

- `flat + auto`
  作为“大类识别 + 槽位填充”主链路默认值
- `hierarchical + auto`
  作为“大类 -> 叶子 -> 槽位”增强链路
- `always`
  只保留给多意图、复杂条件、专项回归验证场景

## 13.2 主要风险

1. 代码模块数会变多  
   这是必要复杂度，不建议继续压回单个 builder。

2. Router 侧会承担更多等待态逻辑  
   但这比把缺槽位理解丢给 agent 更符合职责边界。

3. Admin 需要增加字段  
   但层级路由不正式入模，后续所有 graph 和槽位治理都会继续依赖 prompt。

## 14. 最终结论

后续代码开发应以本方案为准，核心方向是：

1. 从“扁平 intent + 一次性 unified builder”转向“层级路由 + 分段抽取 + 确定性校验”。
2. Graph 保留，但 Graph 只表示叶子执行关系，不表示分类过程。
3. 条件必须成为一等治理对象，独立于执行槽位，但不阻塞首批主链路开发。
4. memory 只作为受控候选源，不能直接当当前轮真值。
5. Router 在下发 agent 前必须完成可执行判断。

如果只允许优先启动最关键的开发项，顺序应为：

1. 意图注册模型补 `domain_code`
2. Catalog 支持按 domain 聚合
3. DomainRouter + LeafIntentRouter
4. SlotExtractor
5. UnderstandingValidator
6. Router pre-dispatch gate
7. 结构化 memory
8. GraphCompiler 与 Runtime 收口
9. 条件专项治理

前 8 项完成后，主链路会先稳定成“层级路由后的叶子执行图运行时”；第 9 项再把条件能力作为独立增强工作流接入。
