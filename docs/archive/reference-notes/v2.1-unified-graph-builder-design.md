# V2.1 Unified Graph Builder 设计

## 1. 目标

V2 当前首轮新消息的 Router 侧流程是：

1. `IntentRecognizer` 做多意图识别
2. `IntentGraphPlanner` 再做一次图规划

这条链路已经能跑，但有两个明显问题：

- 首轮建图至少要过 2 次 LLM
- “识别”和“建图”之间只共享 `intent_code + confidence`，对 slot 约束、单动作边界、节点拆分规则的传递不够强

V2.1 的目标不是推翻 V2，而是把首轮“识别 + 建图”收敛成一次统一的 LLM 调用，让模型在同一个输出里同时给出：

- `primary_intents`
- `candidate_intents`
- `nodes`
- `edges`
- `needs_confirmation`

然后由后端的确定性代码做验证、归一化和兼容兜底。

## 2. 为什么要合并

合并有明确理论依据。

当前真正决定 graph 质量的，不只是 “用户命中了哪些 intent”，而是：

- 哪些词只是某个 intent 的槽位
- 哪些片段代表新的独立业务目标
- 某个 intent 是否允许重复节点
- 某个节点是否需要确认
- 条件依赖是否成立

这些判断天然不是两段孤立推理，而是同一个语义决策过程。

如果先只产出平面的 `intent list`，再把信息交给第二次 planner，planner 已经丢掉了很多一手判断上下文。尤其在下面这些句子里，损失会更明显：

- “我要给我弟弟转500”
- “帮我查一下余额，如果超过5000就给我媳妇儿转1000”
- “先给我媳妇儿转500，再给我弟弟转300”

因此 V2.1 采用：

- **一次 LLM 直接产出 GraphDraft**
- **后端再做 deterministic normalization**

这比“识别一次 + 规划一次”的语义闭环更完整。

## 3. 为什么暂时不合并 Turn Interpreter

`waiting_node` / `pending_graph` 的自然语言处理仍然保留为独立的 `TurnInterpreter`，暂时不并进 unified builder。

原因很简单：

- 首轮建图和等待态解释，不是同一个问题
- 等待态更依赖当前 graph / waiting node / pending graph 的上下文
- 这里还涉及取消、恢复、修改、重规划等强状态语义

如果现在把它也塞进统一 builder，会把“首轮建图”和“图内回合解释”耦到一起，调试会更困难。

所以 V2.1 的边界是：

- **首轮新消息：统一 builder**
- **等待态 / 重规划：沿用 recognizer + turn interpreter + planner**

这是一个有意的分阶段演进，而不是半成品。

## 4. 路由策略

V2.1 不新增 `/chat/v2.1` 或 `/api/router/v2.1`。

保留现有版本入口：

- `/chat/v2`
- `/api/router/v2`

内部通过配置开关控制：

- `ROUTER_V2_GRAPH_BUILD_MODE=legacy|unified`

默认建议：

- 开发联调阶段先用 `legacy`
- 回归通过后再切到 `unified`

这样可以避免再维护一套路由、前端页面、K8s 入口和 SSE 协议。

## 5. 注册期强约束

V2.1 的前提是：意图注册时必须明确“要素约束”。

### 5.1 slot_schema

每个 intent 在注册时新增 `slot_schema`，典型字段：

- `slot_key`
- `label`
- `description`
- `value_type`
- `required`
- `allow_from_history`
- `aliases`
- `examples`
- `overwrite_policy`

它的作用不是给前端看而已，而是直接提供给 unified builder，告诉模型：

- 哪些内容是槽位
- 哪些槽位允许从历史补
- 哪些槽位可以覆盖旧值

这一步是避免“把一个转账意图拆成多个 intent”的关键。

### 5.2 graph_build_hints

每个 intent 还要注册 `graph_build_hints`，用于给 builder 提供建图边界，例如：

- `intent_scope_rule`
- `planner_notes`
- `single_node_examples`
- `multi_node_examples`
- `confirm_policy`
- `max_nodes_per_message`

这些 hint 不是运行时临时拼装的 prompt 文案，而是意图定义的一部分。

## 6. Unified Output

统一 builder 的目标输出是 `GraphDraft`：

```json
{
  "summary": "string",
  "needs_confirmation": false,
  "primary_intents": [
    {
      "intent_code": "transfer_money",
      "confidence": 0.97,
      "reason": "single transfer action"
    }
  ],
  "candidate_intents": [],
  "nodes": [
    {
      "intent_code": "transfer_money",
      "title": "给我弟弟转500",
      "confidence": 0.97,
      "source_fragment": "给我弟弟转500",
      "slot_memory": {
        "recipient_name": "我弟弟",
        "amount": "500"
      }
    }
  ],
  "edges": []
}
```

注意：

- `primary_intents` 是识别结果
- `nodes/edges` 是 graph factory 结果
- 两者来自同一次 LLM 输出

## 7. Deterministic Normalization

LLM 只负责给 draft，不直接决定最终 runtime graph。

V2.1 后端新增 `GraphDraftNormalizer`，负责：

1. 过滤未注册的 `intent_code`
2. 按 intent threshold 归一化 `primary/candidate`
3. 只允许 `primary_intents` 进入可执行 graph
4. 清理非法 edge 引用
5. 按 `confirm_policy` 决定是否进入 `waiting_confirmation`
6. 在 draft 不完整时回落到 legacy planner

这一步很关键。

V2.1 不是“把所有决定都交给大模型”，而是：

- **LLM 负责语义起草**
- **Deterministic layer 负责边界约束**

## 8. 当前落地方案

本轮编码先完成三件事：

1. intent 注册模型增加 `slot_schema` / `graph_build_hints`
2. SQL repository 增加兼容列与自动补列逻辑
3. 新增 `LLMIntentGraphBuilder` 与 `GraphDraftNormalizer`

并且把 unified builder 接到 `GraphRouterOrchestrator` 的可选路径中：

- `legacy` 模式：仍走 `recognizer -> planner`
- `unified` 模式：首轮新消息走 `graph_builder`

等待态多轮交互暂不切换。

## 9. 风险与后续

### 9.1 当前风险

- 首轮建图走 unified，等待态重规划还走 legacy planner，短期内存在“双轨”
- `graph_build_hints` 目前主要用于 prompt 和确认策略，尚未完全做 deterministic enforcement
- Admin 前端还没有把 slot/hints 做成强表单化配置

### 9.2 下一步建议

1. 给 Admin 管理面补充 slot/hints 编辑能力
2. 把 `waiting_node` 下的重规划逐步迁到 unified builder
3. 把 `max_nodes_per_message`、重复节点策略等从 prompt 提示升级为确定性校验
4. 为多 agent / 条件依赖场景增加更多集成测试，而不只是单测

## 10. 结论

结论不是 “LangGraph/LLMCompiler 完全没价值”，而是：

- 当前项目 V2 已经具备 LLMCompiler 所强调的绝大部分 runtime 基础能力
- 真正缺的不是另一套 runtime 壳子，而是更强的 **意图识别 + graph factory 一体化**

因此 V2.1 的正确方向不是引入 `/chat/v2.1` 新系统，而是直接升级现有 `/chat/v2` 的首轮建图能力，并把注册期要素约束正式收进系统模型。
