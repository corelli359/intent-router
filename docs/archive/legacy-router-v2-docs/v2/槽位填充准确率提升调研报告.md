# 槽位填充准确率提升调研报告

调研时间：2026-04-11

## 1. 结论摘要

在不考虑模型微调、传统机器学习、深度学习专项训练的前提下，提升槽位填充准确率，最有效的方向不是继续单纯堆 prompt，而是把当前链路从“LLM 一次性产出槽位”升级为“结构化约束 + 证据绑定 + 校验修复 + 受控记忆预填充”的组合方案。

结合当前仓库实现与外部资料，优先级最高的改造建议是：

1. 把 Router 侧从“提示词要求输出 JSON”升级为“严格 schema 约束的结构化输出 / function calling”。
2. 把槽位填充改成两阶段：先抽取，再做确定性校验与定向修复，而不是一次性相信首轮输出。
3. 强化 `slot_schema`，尤其是 `field_code`、`role`、`examples`、`counter_examples`、`normalization_hint`、`confirmation_policy`。
4. 所有槽位都尽量保留证据来源，至少记录 `source`、`source_text`、`confidence`，高风险场景要求“有证据才落槽位”。
5. memory 有帮助，但只能做“受控候选预填充”，不能做“无条件自动注入”。

我基于资料的核心判断是：

- 对稳定、低变化、可长期复用的槽位，memory 通常有帮助。
- 对金额、日期、订单号、当前动作对象、条件阈值等强时效槽位，memory 很容易引入误填和错绑。
- 对当前仓库，memory 最适合扮演“候选来源 + 需要确认的预填充值”，而不应该被直接当成当前轮真值。

## 2. 本次调研范围

本报告只讨论以下方案：

- 基于大模型的结构化输出
- prompt / schema / workflow 设计
- 记忆注入、检索、校验、澄清
- 确定性规则校验、归一化、证据绑定
- 多轮对话中的槽位补全

明确不讨论：

- 模型微调
- 传统机器学习分类器或序列标注器
- 额外训练专用深度学习槽位模型

## 3. 当前仓库已有基础能力

从当前代码看，仓库已经具备一部分正确方向上的能力，不是从零开始。

### 3.1 已有能力

1. 已有显式槽位 schema。
   - `backend/contracts/intent-registry/src/intent_registry_contracts/models.py`
   - 已支持 `slot_key`、`field_code`、`role`、`value_type`、`allow_from_history`、`confirmation_policy`、`overwrite_policy` 等。

2. 已有槽位 grounding。
   - `backend/services/router-service/src/router_service/core/slot_grounding.py`
   - 当前会对数字类和值类做基础文本落地校验，避免部分“凭空补槽位”。

3. 已有历史槽位受控回填。
   - `backend/services/router-service/src/router_service/core/slot_resolution_service.py`
   - `allow_from_history=true` 的槽位才允许从历史注入。

4. 已有来源标注。
   - `backend/services/router-service/src/router_service/core/v2_domain.py`
   - 当前节点支持 `slot_bindings` 与 `SlotBindingSource`，这是后续做审计、确认和修复的关键基础。

5. 已有“历史复用后强制确认”的机制。
   - `backend/services/router-service/src/router_service/core/slot_resolution_service.py`
   - `backend/services/router-service/src/router_service/core/v2_graph_builder.py`
   - 这点方向是对的，尤其适合金融类高风险动作。

### 3.2 现阶段主要短板

1. Router 侧还没有真正用上严格结构化输出。
   - `backend/services/router-service/src/router_service/core/llm_client.py`
   - 当前 `run_json()` 仍是流式取文本后再 `extract_json_value()`，本质上更接近“JSON mode + 解析”，不是严格 schema 约束。
   - `Settings.llm_structured_output_method` 已存在，但 Router 代码路径没有真正把它用起来。

2. 长期记忆仍然偏“字符串化”。
   - `backend/services/router-service/src/router_service/core/memory_store.py`
   - 目前 `long_term_memory` 主要是 `"intent_code: key=value"` 以及最近消息文本，缺少类型、时间、确认状态、TTL、冲突版本等结构化信息。

3. grounding 还比较轻。
   - 当前更偏 substring / digits containment。
   - 对金额、日期、枚举、相对时间、别名、多候选冲突、否定修正等场景还不够强。

4. 证据绑定还没有升级成强约束。
   - Prompt 已要求尽量输出 `slot_bindings`，但运行时对“没有证据的槽位是否允许落到 `slot_memory`”还不够严格。

5. 缺少独立的“抽取后校验/修复”回路。
   - 当前主要是一次生成、一次归一化。
   - 对于复杂句式、跨轮修正、同类多槽位竞争，准确率会被单次生成上限限制。

## 4. 外部调研结论

### 4.1 结构化约束比“只让模型输出 JSON”更可靠

OpenAI 在 2024-08-06 发布的 Structured Outputs 说明中明确区分了 JSON mode 与严格 schema 约束：

- JSON mode 只能保证输出是合法 JSON，不保证符合指定 schema。
- Structured Outputs 在 `strict: true` 下，目标是让输出严格匹配开发者给定的 JSON Schema。

同一篇官方文章还给出了复杂 JSON schema 跟随能力的评测结果：开启 Structured Outputs 后，`gpt-4o-2024-08-06` 在其复杂 schema eval 上达到 100% 可靠性，而旧模型显著低得多。

这对槽位填充的直接启发是：

- 槽位提取首先是“结构化生成”问题，不只是“理解问题”。
- 如果 schema 层没有锁住，后续 prompt 再细也会被格式漂移、漏字段、错字段拖累。

这和 DST / slot filling 方向的研究结论一致。`FnCTOD`（ACL 2024）把零样本 DST 改写成 function calling 问题，报告中指出该方法可以在不做额外模型调优的情况下提升零样本 DST，并使 GPT-3.5 / GPT-4 的平均 JGA 分别提升 4.8% / 14%。`ParsingDST`（EMNLP Findings 2023）则把 DST 改写成 text-to-JSON 的 semantic parsing，并报告优于既有 zero-shot ICL 方法。

结论：

- 对当前项目，第一优先级不是重写 prompt，而是先把 Router 侧识别和图构建都切到真正的结构化输出接口。

### 4.2 “抽取一次就执行”不如“抽取 + 校验 + 修复”

`SCIR`（2025-12 提交）提出了 schema-based 的自纠错迭代式信息抽取框架。它的核心思想不是训练一个新模型，而是在现有 LLM / IE 系统之上增加双路径自纠错和反馈驱动优化。论文报告在多个 IE 任务上取得平均 5.27% 的 span-based Micro-F1 提升。

虽然该工作不是专门针对任务型对话，但它对槽位填充非常适用，因为槽位填充本质上也是受 schema 约束的结构化信息抽取。

这带来的工程结论是：

1. 第一轮负责“召回候选槽位”。
2. 第二轮不重新开放生成，而是只检查：
   - 是否落在合法 slot key
   - 是否符合 value type
   - 是否与 source text 一致
   - 是否与其它槽位冲突
   - 是否触发 history / confirmation 规则
3. 只对失败项做定向修复或澄清，不重跑整张图。

### 4.3 证据化抽取优于纯黑盒抽取

`TrialMind`（2024）虽然面向临床证据抽取，但其结果对本项目同样有参考价值。该工作不是只让模型给最终结构化结果，而是把搜索、筛选、抽取做成面向 human-AI collaboration 的流程。论文报告在数据抽取上相对 GPT-4 baseline 有 29.6% 到 61.5% 的提升。

对当前项目的启发不是“照搬医疗流程”，而是：

- 槽位值如果能绑定到原始证据片段，后续确认、修复、审计都会明显更稳。
- 证据化抽取特别适合处理“两个金额”“两个姓名”“阈值金额 vs 执行金额”这类错绑问题。

这和仓库中已经存在的 `slot_bindings.source_text` 方向一致，说明当前设计是对路的，只是还不够强制。

### 4.4 memory 确实有帮助，但不能粗暴注入

关于“注入 memory 有没有帮助”，外部资料给出的结论是“有帮助，但收益与副作用同时存在”。

`ES-MemEval`（WWW 2026）评测了长时对话里五类记忆能力：信息抽取、时间推理、冲突检测、弃答、用户建模。其结论包括：

- 显式长期记忆对减少 hallucination、支持个性化是必要的。
- 检索增强（RAG）能提升事实一致性。
- 但在时间变化和用户状态演化上，现有 RAG 仍然薄弱。

`OP-Bench`（2026-01）进一步指出：

- 引入 memory 后，过度个性化是普遍问题。
- 代理会在不必要时仍然检索并过度关注用户记忆。
- 论文因此提出了轻量的 memory filtering 机制。

因此，对本项目更准确的结论不是“要不要 memory”，而是：

- 要用 memory，但必须加 gating、过滤、冲突检查和确认策略。
- memory 适合做“候选来源”，不适合直接替代当前轮证据。

## 5. 对 memory 的明确判断

## 5.1 哪些槽位适合 memory 预填充

适合从历史复用的，通常是稳定且用户不希望每次重填的槽位：

- 默认银行卡号
- 手机号后四位
- 常用地址
- 默认币种
- 稳定联系人称谓
- 用户长期偏好

这类槽位共同特征是：

- 生命周期长
- 单次对话中不频繁变化
- 当前轮未明确给出时，历史值仍有较大概率成立
- 即使预填，也适合提示用户确认

## 5.2 哪些槽位不适合直接从 memory 注入

不建议直接历史注入的槽位包括：

- 本次交易金额
- 本次执行日期 / 时间
- 当前动作对象
- 订单号 / 账单号 / 流水号
- 条件阈值
- 与本轮上下文强绑定的临时参数

这类槽位的共性是：

- 强时效
- 高频变化
- 与“本轮语义”强绑定
- 错一次代价高

对这类槽位，即使历史里有值，也最多作为候选参考，不应默认落进执行态 `slot_memory`。

## 5.3 对当前项目的 memory 设计建议

当前项目已经有：

- `allow_from_history`
- 历史注入后进入确认态

这两点应保留，但建议升级为以下机制：

1. 把 long-term memory 从字符串改成结构化记录。
   - 建议字段：`slot_key`、`value`、`value_type`、`source_intent_code`、`source_session_id`、`confirmed`、`updated_at`、`ttl`、`sensitivity_level`。

2. 历史检索只返回“候选槽位”，不直接覆盖当前轮。
   - 当前轮文本命中优先。
   - 历史值只能补空，不能压掉当前轮显式值。

3. memory 检索要加过滤条件。
   - 只查 `allow_from_history=true`
   - 只查 `confirmed=true`
   - 只查未过期数据
   - 同一槽位若有多个候选，需要做冲突消解或直接要求确认

4. 对敏感动作保持“注入即确认”。
   - 当前仓库已经这样做，方向正确，不建议放宽。

5. 把 memory 当成 side input，而不是 prompt 主体。
   - 不建议把大量历史文本直接塞进 prompt。
   - 更推荐先做 retrieval，再把极少量结构化候选传给抽取器 / 校验器。

基于上述资料，我的判断是：

- memory 对“补齐稳定槽位的召回率”有帮助。
- memory 对“整体准确率”只有在加入过滤、冲突检测、确认态后才是净收益。
- 如果不做这些控制，memory 反而会抬高误填率和错绑率。

## 6. 推荐方案与优先级

## 6.1 P0：立即做

### P0-1 Router 侧切换到严格结构化输出

目标：

- 让 `recognizer`、`unified graph builder`、`turn interpreter`、`recommendation router` 都走严格 schema。

建议改造点：

- `backend/services/router-service/src/router_service/core/llm_client.py`
- `backend/services/router-service/src/router_service/core/recognizer.py`
- `backend/services/router-service/src/router_service/core/v2_graph_builder.py`
- `backend/services/router-service/src/router_service/core/v2_planner.py`
- `backend/services/router-service/src/router_service/core/v2_recommendation_router.py`

具体建议：

- 为各调用点显式传入 Pydantic schema。
- 优先支持 `json_schema` 或 `function_calling` 的 strict 路径。
- 只有结构化调用失败时，才回退到文本 JSON 解析。
- 可直接参考 agent 侧已有实现模式，例如 `backend/services/agents/transfer-money-agent/src/transfer_money_agent/support.py` 中的 `with_structured_output()` 路径，避免 Router 另起一套完全不同的结构化调用接口。

预期收益：

- 降低格式错误、错字段、漏字段、额外字段。
- 让后续 validator 的输入更稳定。

### P0-2 增加“槽位校验与修复”层

建议新增一个类似 `slot_validation.py` 的模块，职责包括：

- `slot_key` 合法性检查
- `value_type` 检查
- required / optional 检查
- `allow_from_history` 检查
- 当前轮 grounding 检查
- 多槽位冲突检查
- 高风险槽位确认策略检查

建议流程：

1. LLM 抽取 `slot_memory` + `slot_bindings`
2. validator 打标签：
   - valid
   - invalid
   - ambiguous
   - conflict
3. 只对 invalid / ambiguous / conflict 的槽位触发 repair prompt 或澄清提问

预期收益：

- 对复杂句式和多槽位竞争场景提升明显。
- 减少“整张图重建”导致的二次偏移。

### P0-3 把 `slot_bindings` 从“可选元数据”提升为“核心质量机制”

建议规则：

1. 来自当前轮的槽位，尽量必须有 `source_text`。
2. 没有证据片段的高风险槽位，不进入可执行 `slot_memory`。
3. 历史注入槽位必须显式标记 `source=history`。
4. 推荐默认值必须显式标记 `source=recommendation`。

当前仓库已有基础：

- `slot_bindings`
- `SlotBindingSource`

建议增强点：

- 对金额、日期、账号、手机号、条件阈值等类型增加强证据要求。

### P0-4 强化 schema 治理，而不是只强化 prompt

当前 `IntentSlotDefinition` 已经支持很多字段，但建议真正用起来：

- `examples`
- `counter_examples`
- `format_hint`
- `normalization_hint`
- `validation_hint`
- `prompt_hint`

建议重点治理：

1. 同类型多槽位时，必须有角色区分。
   - 例如 `amount` 不能只靠类型区分，必须用 `role` 区分“执行金额”“条件阈值”“账单金额”。

2. 闭集槽位尽量枚举化。
   - 币种、账户类型、操作类型、渠道类型等可闭集化的槽位应尽量提供候选值或 enum。

3. 必须给反例。
   - `counter_examples` 对降低错绑很重要，尤其是金额、姓名、账号类。

## 6.2 P1：第二阶段做

### P1-1 把 memory 从“文本 recall”升级为“结构化候选库”

建议替换当前：

- `memory_type=session_message`
- `memory_type=task_slot_memory`
- `content="intent: key=value"`

为更结构化的数据：

```json
{
  "slot_key": "card_number",
  "value": "6222021234567890",
  "value_type": "account_number",
  "source_intent_code": "query_account_balance",
  "confirmed": true,
  "updated_at": "2026-04-11T09:00:00Z",
  "ttl_seconds": 2592000,
  "confidence": 0.98
}
```

理由：

- 这样 memory 可以先检索、再过滤、再排序，而不是先拼到 prompt 里让模型自己猜。

### P1-2 引入 targeted clarification，而不是低置信时硬猜

当出现以下情况时，推荐直接补问：

- required 槽位缺失
- 当前轮与历史冲突
- 同一句里存在两个候选值且无法判定绑定关系
- 识别到否定、改口、修正语气
- 历史值可复用但该动作高风险

补问模板要足够短，只问必要槽位。例如：

- “本次转账金额是 500 还是 5000？”
- “你这次要查的是上次那张卡吗？”

我的判断是：

- 在金融场景，少量澄清通常比错误自动填充更优。
- 目标不是“零追问”，而是“避免无根据补槽位”。

### P1-3 做 value-type 级归一化

当前 `slot_grounding.py` 已做了基础字符落地，但还可以继续增强：

- 中文数字转阿拉伯数字
- 币种别名统一
- 相对时间归一化
- 日期格式统一
- 手机号 / 卡号脱敏匹配
- 人名称谓归一化
- 多个 alias 的 canonical mapping

建议把“抽取”和“归一化”分开：

- 抽取负责找值
- 归一化负责变成可执行 canonical value

## 6.3 P2：作为持续治理能力建设

### P2-1 建立专门的槽位回归集

建议覆盖以下 case：

- 单意图多槽位
- 多意图同类槽位竞争
- 条件金额与执行金额共存
- 当前轮修正历史值
- 否定句
- 省略句
- 指代句
- 两个以上同类型值
- 历史可复用但不应复用
- 推荐默认值被用户修改

### P2-2 建立 memory A/B 实验

建议至少跑四组：

1. 无 memory
2. 有 memory，但仅检索不注入
3. 有 memory，受控预填充
4. 有 memory，受控预填充 + 自动确认

重点看：

- 召回率有没有提升
- 错填率有没有上升
- 高风险动作误注入率是多少

我预期第 3 组通常最优，第 4 组风险最高。

## 7. 推荐的目标架构

建议把槽位填充链路收敛为下面这条：

1. Intent 识别
2. 图构建 / 节点抽取
3. 第一轮槽位抽取
4. 结构化 validator
5. history / recommendation / context 候选检索
6. 槽位修复或澄清
7. 归一化与最终落槽位
8. 高风险确认
9. 下发 agent

其中第 4 步和第 5 步最关键：

- 第 4 步保证“不乱填”
- 第 5 步保证“能补齐”

只做第 5 步而不做第 4 步，通常会让 memory 变成误填放大器。

## 8. 适配当前仓库的落地路线

### 第一阶段

目标：先把基础准确率抬上去。

建议事项：

1. Router 接入 strict structured output
2. 新增 slot validator
3. 强制高风险槽位提供 `slot_bindings.source_text`
4. 对现有测试补充“错绑”和“历史误注入”回归样例

### 第二阶段

目标：让 memory 从“可用”变成“可控”。

建议事项：

1. 重构 `LongTermMemoryStore`
2. 引入结构化 memory record
3. 为每个槽位增加 TTL / confirmed / sensitivity
4. 在检索前做 slot-level gating

### 第三阶段

目标：把复杂多轮场景稳定下来。

建议事项：

1. 引入 repair prompt
2. 引入 targeted clarification
3. 引入更细的归一化器
4. 增加离线评测脚本和对比报表

## 9. 建议跟踪的指标

建议不要只看“整句是否成功”，而要单独看槽位指标：

1. `slot_exact_match`
   - 最终槽位值是否与标注完全一致

2. `slot_grounded_rate`
   - 最终槽位是否能在当前轮证据或允许的历史证据中找到依据

3. `history_injection_precision`
   - 被历史注入的槽位里，真正正确的比例

4. `history_leak_rate`
   - 不该从历史注入却被注入的比例

5. `clarification_rate`
   - 为避免误填而触发追问的比例

6. `clarification_success_rate`
   - 追问后槽位正确补齐的比例

7. `slot_repair_success_rate`
   - validator 判错后，repair 成功纠正的比例

8. `execution_blocked_by_missing_slot_rate`
   - 因缺槽位导致无法执行的比例

理想目标不是把 `clarification_rate` 压到最低，而是把 `history_leak_rate` 和高风险错填率压到最低。

## 10. 最终建议

如果只允许做少量高收益改造，我建议按下面顺序做：

1. 先把 Router 改成 strict structured output。
2. 再补 slot validator + repair loop。
3. 再把 memory 改成结构化候选预填充，而不是文本注入。
4. 最后补 targeted clarification 和更强归一化。

对“memory 有没有帮助”的最终回答是：

- 有帮助，但只对一部分槽位有帮助。
- 对当前项目，正确用法不是“把 memory 塞给模型”，而是“把 memory 作为受控候选源”。
- 如果没有 gating、冲突检测、TTL、确认态，memory 很可能降低而不是提升槽位填充准确率。
